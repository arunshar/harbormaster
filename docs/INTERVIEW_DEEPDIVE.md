# Harbormaster interview deep-dive: stories, numbers, traps

Written 2026-07-04, for me alone, night-before and morning-of prep for MLE, ML-infra, platform, and FDE loops. Everything below is answer-shaped and speakable in 60 to 120 seconds per block.

## 1. The 60-second story

I built Harbormaster, a personal maritime anomaly-detection platform on AWS, to close resume gaps I named honestly in a Supabase Multigres cover note: change-data-capture, streaming, production distributed systems, and MLOps observability. It runs under a $75-a-month hard cost cap I built before I built anything else. Phases 0, 1, and 3 have run live in AWS and Phase 3 has already been torn down clean. Phase 2's CDC pipeline is accepted end to end on a local stack; its AWS showcase hasn't run yet, and I say that plainly. Phase 4, a drift-to-HITL-to-RL flywheel, is code-complete and I red-teamed its own safety mechanism twice, finding and fixing a real gap the same day it was found. Phase 5, multi-tenant EKS and a Bedrock explanation layer, is authored and signed off, the build is a later sprint. I did this with three layers of review on top of the code itself: internal adversarial passes, an external audit against AWS's own canonical examples, and live runs that found things no review could. What it does not prove: I did not build a sharded query router and I did not implement a consensus protocol. That is Vitess and Multigres territory, and I say so before anyone asks.

## 3. Architecture walkthrough, the 45-minute version

Say the two-plane split first: heavy training happens on MSI, free H100s, off the cloud entirely; serving happens on AWS. Local fakes and injected I/O prove the same code path the AWS demo windows exercise for real, so a reviewer never has to take my word that the logic is right, only that the AWS wiring around it is.

**Broker split.** Kinesis carries the telemetry stream (AIS positions), Kafka carries the CDC stream (registry changes). Different reliability shapes, different consumers, no reason to force one broker to do both jobs.

**CDC versus an outbox.** I used log-based CDC over logical replication rather than a transactional outbox table, because Postgres already emits a durable, ordered change log through `wal_level=logical`, and an outbox would be reinventing that inside application code for no reliability gain.

**Idempotent receiver.** Every write carries a monotonic LSN; the sink's conditional expression rejects anything older than what it already applied. That is what makes replaying the entire Kafka topic safe: duplicate transport, zero duplicate state.

**The promotion saga, DR-3.** Shadow, then canary at 5, 25, 50, 100 percent, with a burn-rate breach triggering a full, immediate revert as the compensating transaction. I concretized this using SageMaker's own native production-variant weighting rather than Argo CD, since only the Pi-DPM head sits behind a managed endpoint through Phase 4; a real front door on EKS in Phase 5 is where Argo CD becomes the right tool again, and the decision record says so.

**SLOs and burn-rate, DR-13.** I'll be honest here if pushed: this decision was cited from Phase 3 onward, but the actual burn-rate calculator did not exist anywhere in the repo until Phase 5's fact-check went looking for it and found only three static thresholds. Every promotion run before that supplied a fake or a hardcoded false. I'd rather say that than let someone assume the mechanism was real earlier than it was.

**Backpressure and the physics gate, DR-6.** Flink's event-time windows and watermarks let backpressure propagate naturally when a downstream stage slows. A cheap physical-plausibility gate runs first and sheds anything that cannot possibly be anomalous before the expensive model ever sees it.

**Partitioning and hot shards, DR-9.** Kinesis is sharded by vessel MMSI; the hot-shard risk (one vessel dominating a partition) is a named, anticipated war story, P1, not yet grounded, because the fixture-scale traffic hasn't produced a real hot shard yet.

**CQRS, DR-4.** Postgres is the write model for the registry and operational state; Feast, DynamoDB, and a Redis cache are the read models the serving path actually queries. I'll say directly why this isn't over-engineering for a personal project: the write side wants relational integrity for a registry a human edits through a console, the read side wants single-digit-millisecond lookups at serving time, and those are different enough workloads that forcing one store to serve both would compromise one side or the other. The CDC pipeline is what keeps the read side honest.

**Multi-tenant isolation as a bulkhead, DR-7.** Every record and request will carry a `tenant_id`, default-deny: a request with no matching tenant grant gets rejected outright, not silently scoped to nothing. This decision was written before Phase 5 existed to implement it; Phase 5 is where it becomes real code, enforced at the database layer through row-level security rather than an application-layer check that a future refactor could accidentally bypass.

## 4. STAR bank A: grounded platform war stories

Eighteen stories, P9 through P26, one after another, each compressed. Source for every one: `PLATFORM_WAR_STORIES.md`. I do not soften the wrong hypotheses; that beat is the point.

**P9, replication-slot bloat.** Situation: a stalled CDC consumer. Action: watched `pg_replication_slots` lag climb from 0 to 74,035,000 bytes across five write rounds while the consumer sat dead, confirmed the alert fires at the configured threshold, restarted the consumer, watched lag drain back to zero. Wrong first hypothesis: none needed, this was the anticipated failure mode, drilled on purpose. Result: the alert and the drain both work as designed. Follow-ups: "Why does a stalled consumer matter for disk, not just staleness?" Because a replication slot is a lien on the source database's own disk; Postgres cannot vacuum WAL the slot hasn't confirmed flushing, so a stalled consumer eventually fills the source's storage, not just falls behind. "What's the actual alarm threshold?" 200 megabytes by default, tunable per deployment.

**P10, duplicate CDC events after a restart.** Situation: an at-least-once consumer restarted mid-batch. Action: compared a guarded run against a `--no-guard` drill flag; the no-guard run visibly regressed a severity score from 0.95 to 0.90 under a specific zombie-rebalance schedule, the guarded run converged to a byte-identical final state under every schedule tested. Wrong first hypothesis: a simple retry-safe upsert is enough for idempotency. Root cause: order matters, not just repetition; an upsert without an LSN guard can apply an older redelivered value over a newer one if the redelivery lands after a later legitimate update. Result: the LSN-guarded conditional write. Follow-ups: "Why not just dedupe by event ID?" Because dedup by ID stops a literal repeat but does nothing about an out-of-order repeat, which is the actual failure mode a rebalance produces. "How did you find the specific regressing schedule?" A property test shuffling delivery order across many seeds, not a hand-picked example.

**P11, training-serving skew.** Situation: a candidate model whose offline feature encoding differed from its online encoding. Action: the holdout gate scored it at AUC 1.0, a clean pass; the shadow comparator, run against the same batch through the real online path, caught a mean absolute divergence of 0.6056 against a 0.05 threshold. Wrong first hypothesis: a strong offline AUC is sufficient evidence a candidate is safe to promote. Root cause: the holdout set is built from the same offline-encoded pipeline the skew itself came from, so it structurally cannot see an offline-online mismatch; only real online traffic can. Result: shadow exists precisely because holdout is necessary but not sufficient. Follow-ups: "Isn't that just a training-serving skew bug you could've caught with better tests?" The point of the drill was proving the *architecture* catches it, not that any one test would have; shadow is the layer designed for exactly this blind spot. "What would you do differently next time?" Add an explicit encoding-parity check between the offline export and the online feature path as a fast, cheap pre-check before the expensive holdout run.

**P12, canary regression at real traffic.** Situation: a candidate that passed both holdout and shadow regressed only once real canary traffic reached it. Action: the promotion state machine advanced to weight 5, a burn-rate check fired at weight 25, `revert_to_champion` executed immediately, weight never advanced to 50 or 100. Wrong first hypothesis: if holdout and shadow both pass, canary is a formality. Root cause: shadow only samples a fraction of live traffic; a rare-but-real distribution can pass through an entire shadow window by chance. Result: canary at increasing, bounded weights is the layer that meets the *full* traffic distribution before a rollout is irreversible. Follow-ups: "How is the revert guaranteed atomic?" It is a single action, not a stepped rollback, tested at all four canary weights independently. "What's the actual burn-rate formula?" The Google SRE multi-window pattern, cited in DR-13, not yet implemented for real until Phase 5.

**P13, PyFlink's worker is not the driver's environment.** Situation: a stateful Flink operator failing `ModuleNotFoundError` for a package that imported fine at driver startup. Action: traced it to Beam's process-mode portability harness running the operator in a separate Python subprocess that does not inherit the driver's `sys.path`; two attempts to ship the code as an explicit Flink dependency both hit real, confirmed Managed Flink staging bugs before the actual fix. Wrong first hypothesis: if the driver can import it, the job can use it anywhere. Result: inlined the logic directly into the entry-point module so it serializes by value, no staging step at all; used `env.set_python_requirements()` for the one genuine third-party dependency. Follow-ups: "Why not just fix the pyFiles bug?" It was a real, deterministic bug in the platform's own dependency-staging subsystem, not something I could patch from application code; inlining sidesteps the whole mechanism. "How did you confirm it was cloudpickle's by-value-versus-by-reference rule and not something else?" By testing that code defined in `__main__` worked while the identical code imported from a package did not, isolating the variable directly.

**P14, TTL fires on arrival for replayed data.** Situation: DynamoDB items disappearing seconds after being written, with zero errors anywhere. Action: found the item count fluctuating on repeated scans, traced it to a TTL attribute computed from the AIS fix's own event time; the replay fixture's 2024 timestamps meant every item was already a year past its own expiry the instant it landed. Wrong first hypothesis: a stalled Kinesis consumer, spent real time ruling that out first via CloudWatch metrics. Result: overrode TTL with a wall-clock value at the write site, left the shared, unit-tested feature function's real production formula untouched. Follow-ups: "Why didn't unit tests catch this?" Because unit tests use synthetic, sensible timestamps; this only appears when replaying genuinely historical data through a live pipeline, a demo-specific interaction. "What's the general lesson?" Any downstream deadline math derived from a payload's own timestamp needs a second look whenever that payload might be historical, not just live.

**P15, a schema drifted behind a swallowed exception.** Situation: the HITL queue staying empty despite a fully working upstream pipeline. Action: read the serving API's own access logs, not just Flink's, and found every scorer call returning 422, silently, because the call site wrapped the network request in a bare `except: pass`. Wrong first hypothesis: the pipeline works, nothing anomalous enough is happening to flag. Root cause: the payload sent a stale schema key neither side's tests caught, because the two components live in different subsystems with independent suites. Result: restructured the payload to the real schema, updated the unit test to assert the correct shape. Follow-ups: "How do you prevent this class of bug generally?" A contract test that exercises both sides against the same schema definition would have caught it; that's a real gap this fix didn't close. "Why was the exception swallowed in the first place?" A best-effort design choice for a non-critical scorer call, reasonable in isolation, but it meant a hard failure produced zero signal anywhere a developer would look.

**P16, a planner silently skipped a detector.** Situation: a documented anomaly never reaching HITL despite reaching the scorer successfully. Action: traced a deterministic planner's routing rule, which only adds the gap detector to the execution plan once history length reaches three; the upstream Flink job only ever supplied one prior fix. Confirmed by sending the exact expected payload directly via curl with five history entries, which correctly returned the anomaly. Wrong first hypothesis: the detector's own severity threshold must not be tuned for this case. Result: changed Flink's keyed state to a rolling five-entry window. Follow-ups: "Why didn't tests catch a whole detector never running?" Because "it's scoring successfully" and "it's being evaluated for X" are different claims, and nothing directly asserted the second one for real traffic shapes. "What's the transferable lesson?" When a routing layer conditions on an input's shape rather than its values, verify the shape assumption directly with a synthetic payload, don't infer it from downstream success.

**P17, an EMR job's own IAM role couldn't read its own code.** Situation: the first-ever live EMR job failing `FileNotFoundException` for a file confirmed to exist via a direct CLI check. Action: found the job-execution IAM policy granted read on the input and output prefixes but never on the code prefix where the entry point itself lives. Wrong first hypothesis: the upload silently failed. Result: added a scoped `ReadJobCode` IAM statement. Follow-ups: "Why does Spark report 'file not found' instead of 'access denied'?" Because IAM denies at the S3 API level and Spark's error handling surfaces that generically, which is exactly what made the wrong hypothesis plausible. "How do you avoid this in future modules?" Write the least-privilege policy against every resource the job actually touches at runtime, including its own packaging artifacts, not just its data inputs and outputs.

**P18, a 3.10-only idiom broke on the real runtime.** Situation: a `zip(strict=True)` call failing on EMR after two earlier fixes in the same live run. Action: confirmed the dev machine runs Python 3.12 but EMR's actual image ships 3.9.21, by pulling the real image and checking directly, not assuming parity. Result: removed the keyword with an explicit lint suppression rather than weakening the project-wide rule that requires it elsewhere. Follow-ups: "Why keep the strict= rule at all if it broke here?" Because it's correct and valuable for every file that runs on the dev machine's own Python; the fix scopes the exception to exactly the three EMR-only call sites. "How do you prevent this happening again on a different managed service?" Pull and check the real runtime's version directly before assuming feature parity, every time a job targets a vendor-controlled image.

**P19, a schema declaration didn't match its own function's contract.** Situation: a Spark type error on a column whose transformation function's own docstring already described the correct output type. Action: traced a `mapInPandas` call declaring the raw input schema as its output schema, when the actual function coerces a string column to a timestamp. Result: defined a separate output schema matching the function's real contract. Follow-ups: "How did the docstring predict the bug?" It stated outright what the transformation does; the Spark wiring around it simply never got checked against that stated contract when the schema was declared. "What's the review takeaway?" A typed boundary's declared schema should be checked against the function's actual return shape, not the shape of what was fed in.

**P20, a catalog client needed its own region source.** Situation: `NoRegionError` on a job running inside `us-east-1` with the region variable set everywhere else in the process. Action: read pyiceberg's own source and found its Glue catalog client resolves region strictly from two specific catalog-property keys, never from ambient environment variables or another client's configuration. Result: added those two exact keys. Follow-ups: "Why didn't the ambient AWS_REGION work?" Because this specific client's configuration surface is narrower than "however AWS clients normally get a region," a library-specific behavior, not a general AWS SDK rule. "How do you debug a config error that 'shouldn't happen'?" Check whether the specific failing client actually reads any of the sources you assume it does, rather than assuming the surrounding environment is sufficient.

**P21, a modern Docker build produced a manifest SageMaker rejects.** Situation: `CreateModel` failing on an unsupported manifest media type for an image built with an ordinary `docker build`. Action: found that Docker Desktop's legacy builder is effectively gone and the old `DOCKER_BUILDKIT=0` escape hatch no longer changes the output format at all; verified the real pushed manifest type directly with `docker manifest inspect` rather than trusting a build flag. Result: an explicit buildx output configuration forcing the older Docker v2 manifest format. Follow-ups: "Why not just downgrade Docker?" That trades a real problem for an environment fragility; forcing the correct output format is portable and explicit. "How do you know the fix actually worked before spending another apply cycle?" Verified the pushed manifest's media type directly before touching Terraform again.

**P22, a container with no ENTRYPOINT couldn't answer SageMaker's own convention.** Situation: `CannotStartContainerError` on an image that ran fine under a bare `docker run`. Action: found the Dockerfile had only a `CMD`, and Docker's own rule is that any argument passed to `docker run <image> <args>` replaces `CMD` entirely rather than appending to it; SageMaker invokes every real-time container as `docker run <image> serve`. Result: a one-line `entrypoint.sh` as the real `ENTRYPOINT`, verified locally with the exact invocation SageMaker uses before the next apply. Follow-ups: "Why didn't the plain docker run catch this?" Because it never passed an argument, so the bug was invisible until the exact target invocation convention was tested. "What's the general lesson?" Test a managed service's documented invocation convention exactly, argument included, not just that the image runs at all.

**P23, a shared git working directory is mutable shared state.** Situation: a clean terraform plan suddenly proposing to revert an already-fixed configuration. Action: recognized this as suspicious rather than assuming the fix was lost, ran `git branch --show-current`, found a different collaborator's branch checkout had silently changed HEAD in the same shared clone. Result: switched back, confirmed no real damage (the plan step is a review, not a mutation, so nothing was actually destroyed). Follow-ups: "How is this different from a normal merge conflict?" A merge conflict is visible and expected; this was invisible, since a checkout in one terminal changes files for every terminal with no notification. "What's the fix going forward?" Use isolated worktrees per collaborator or session rather than a single shared clone.

**P24, a pure state machine outran the real API it drove.** Situation: a real promotion loop succeeding at canary weight 5 then failing at weight 25 with a "cannot update in-progress endpoint" error. Action: found that `UpdateEndpointWeightsAndCapacities` is asynchronous and the endpoint stays `Updating` for real wall-clock time; the loop, correct for a pure synchronous state machine tested against instantaneous fakes, called the next update immediately. Result: added polling for `InService` inside the injected callables, leaving the pure core logic untouched. Follow-ups: "Why didn't the unit tests catch this?" Because they supply instantaneous fakes by construction; this gap only exists at the seam between tested logic and a real asynchronous API. "Where does the fix belong, architecturally?" In the adapter wrapping the real client, never in the tested core state machine.

**P25, an audit finding on paper and a verified live fix are different claims.** Situation: a SageMaker endpoint reporting healthy with zero errors while its scale-in mechanism was structurally unable to ever fire. Action: an external, independent audit against AWS's own canonical examples caught a target-tracking metric spec missing its `EndpointName` dimension; applied the fix, then verified live by reading the real CloudWatch alarms transition out of `INSUFFICIENT_DATA` and confirming a genuine zero-instance scale-down. Result: closed before any real spend accrued. Follow-ups: "Why is 'no errors' not enough evidence a fix works?" Because a dimensionless metric spec silently queries a series that doesn't exist; there is no exception anywhere, only a monthly bill. "What's the interview-ready version of this?" I audited the guardrail protecting my own cost cap hardest, and it paid for itself.

**P26, the same bug class survived its own targeted fix.** Situation: immediately after confirming the scale-in fix worked, a real invocation sent to wake a zero-instance endpoint never triggered the scale-out alarm at all. Action: compared the alarm's Terraform against AWS's own canonical scale-from-zero example and found an extraneous dimension the metric never publishes, then confirmed empirically by querying CloudWatch directly with and without that dimension. Result: removed it, verified live end to end, a genuinely zero-instance endpoint waking to `InService` in about 30 seconds. Follow-ups: "Didn't the audit already fix this exact bug class?" It fixed one instance of it; this is a sibling bug on a different metric the audit itself never checked. "What's the generalizable lesson?" A review that catches a bug class does not certify every other instance of it is safe; the review that catches the most is the one that keeps checking after its own first fix lands.

## 6. The flywheel Q&A

**Why three drift categories instead of one detector?** Because they answer three different questions. Input drift is a population-shape signal. Calibration drift means the model's own predictive uncertainty stopped matching reality. Concept drift means the model started being wrong about cases it used to get right, and production anomaly detection has no ground-truth label stream to check that against directly.

**Why does calibration drift reuse the promotion gate's exact band?** So "drifted out of band" means literally "no longer the model the gate approved," not a separately-invented threshold that could drift out of sync with the thing it's supposed to protect.

**How is concept drift proxied without ground truth?** Near-threshold, high-epistemic-variance trace volume, cross-validated against the rate at which real human reviewers actually disagree with the model. Neither signal alone is trustworthy; the disagreement rate is ground-truth-adjacent but lagging, the variance signal is fast but blind to correctness.

**What's the decision table's false-alarm row?** Rising near-threshold volume with a flat disagreement rate is explicitly not concept drift, just the model seeing harder cases it's still right about. Only a rising disagreement rate, regardless of the other proxy, routes to retraining.

**Why is roughly 90 percent of the response supervised retraining, not RL?** Because calibration drift, the most common real trigger, already has labeled data behind it; there's no reason to reach for a preference-based method when a direct supervised signal exists.

**What does a preference triple actually store, and why per-triple?** Chosen and rejected arms with their full reward breakdowns, plus a flag for whether either arm violated a hard physical constraint. Per-triple, not aggregate, because the platform's central safety claim, that an unbounded penalty term prevents an impossible trajectory from winning, needs to be auditable on every single comparison, not just asserted in the abstract.

**How does the reward-hacking probe work and where does it plug in?** One additional, optional step in the promotion state machine, right after the holdout gate and before shadow. It blocks a DPO or GRPO-sourced candidate when its mean reward rose and either its violation rate or its violation magnitude got worse.

**What did the probe used to miss, and how was it fixed?** A candidate that keeps the same violation count as the honest baseline while making each violation far more severe passed the original rate-only check. Fixed the same day it was found: block on rate rising or mean violation magnitude degrading, whichever fires. Verified: the one existing test whose outcome should have flipped did, correctly, and nothing else changed.

**Why is the preference-triple schema an extension of pi-grpo's real shape, not a copy?** Because pi-grpo's actual `PreferencePair` in production is five fields: a prompt, a chosen arm, a rejected arm, a margin, and a source. Harbormaster's richer schema, with the full reward breakdown and the hard-violation flag attached, is built on top of that real shape rather than replacing it, and the port keeps the real filtering semantics (only rows labeled correct or incorrect, ambiguous excluded) rather than inventing new ones.

**Did building the flywheel change how you think about RL safety generally?** Yes, concretely: I went in assuming a well-designed penalty term is a safeguard by construction. Finding the boundary-riding gap the same day I built the probe changed that to a safeguard is only as good as the adversarial testing run against it, and testing it once with the obvious exploit pattern is not the same as testing it with the exploit pattern its own design docstring already warned about.

## 8. Trap questions, honest answers

**"Is the canary actually shifting traffic?"** No. Single-variant endpoint today, so the weight ramp is currently traffic-inert and the shadow comparison is self-paired. I know this, the audit named it, and a challenger variant is the next real step before that story is fully load-bearing.

**"Did you run this at scale?"** Fixture-scale, live. The full 600 GB historical backfill is costed and designed but hasn't run at full scale yet.

**"Is the model real?"** No, and every response it returns says so. It's a labeled demo stand-in; there's no MSI-trained checkpoint yet. What's real is the infrastructure around it: the async endpoint, the scale-to-zero mechanism, the promotion pipeline, all proven live against this stand-in.

**"Why no Kubernetes?"** Cost, explicitly. An EKS control plane alone runs about $73 a month flat, whether or not anything is running on it, against a $75 cap. I deferred it to Phase 5, which I've now signed off with a structural teardown guard specifically because that non-scale-to-zero cost shape is the platform's biggest single standing risk if it's ever left running.

**"What would break first in production?"** I'd point at the anticipated list, P1 through P8, and say plainly they're anticipated, not experienced: a Kinesis hot shard on one heavily-tracked vessel is the one I'd bet on first, given the partitioning scheme is by MMSI.

**"Are the audit fixes actually applied?"** Yes, both of them, at real commits, verified live against real AWS, not just planned. Seventeen more findings from the same audit remain open and I can name exactly which ones and why they're not urgent.
