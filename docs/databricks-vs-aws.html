Study-to-build bridge

The same platform, two ways: managed Databricks vs hand-built AWS

I studied the production-ML/GenAI lifecycle on managed Databricks, then built the same capabilities AWS-native in Harbormaster. Every managed button in Databricks corresponds to something assembled by hand on AWS. Knowing both is the point.

1 platformDatabricks: one workspace, every step a managed service or button
~15 servicesAWS-native: best-of-breed services wired together under IaC
$75/mohard cap that drives most AWS-native choices

The two arcs in one line each

Databricks production arc

train + register → build agent → RAG → evaluate → serve → monitor, all on the managed platform: MLflow Registry, Unity Catalog, Delta, Model Serving, Lakehouse Monitoring, mlflow.genai judges.

Harbormaster (AWS-native)

hybrid train-on-MSI / serve-on-AWS, under a $75/mo cap, in six phases: 0 FinOps guardrails, 1 production vertical slice, 2 CDC, 3 lake + promotion, 4 drift → HITL → RL flywheel, 5 multi-tenant + FDE.

Honest framing. Harbormaster does not run on Databricks. This is a learning-to-building bridge, not a claim of Databricks production experience. ESRI (company work) and Harbormaster (personal) never merge.

The core difference

Databricks gives you the whole lifecycle as one opinionated, governed platform: the registry, the catalog, the lakehouse, serving, monitoring, and the eval/judge tooling are all integrated and managed. You move fast and you do not run the plumbing.

Harbormaster assembles the same lifecycle from individual AWS services plus an off-cloud GPU plane on MSI. You own the plumbing, but you control cost, you keep the scoring path deterministic and auditable, and you pick the right tool per job. The senior signal is being able to say: "I evaluated the managed pattern and chose the self-hosted equivalent because X."

The five places where I deliberately diverged from the managed default are on the Deliberate divergences page. The per-phase stacks are next.

Phase 0

Foundations & FinOps guardrails

Provision the cost guardrails and infrastructure-as-code substrate before any spend. In Databricks this is account/workspace config; in Harbormaster it is Terraform plus AWS Budgets with a hard deny.

Databricks course: Advanced ML OperationsConcept: IaC, CI/CD, governance, budgets

Databricks (managed)

  • Databricks Asset Bundles (DABs) for IaC
  • Workflows for orchestration
  • Unity Catalog for governance + lineage
  • Cluster / workspace policies; account-level budgets

Harbormaster (AWS-native)

  • Terraform modules: network (VPC), state_stores (S3+DynamoDB), finops
  • AWS Budgets: $30 soft (SNS at $5/$15/$25), $75 hard = budget-action attaches an IAM deny policy to the platform role
  • Cost-anomaly monitor + a teardown Lambda
  • GitHub Actions for plan/apply; providers pinned (aws ~> 5.x)

Phase 1

Production vertical slice: streaming + serving built locally

One genuinely production-grade path: recorded AIS replay → stream → features + physics gate → deterministic scoring front door → anomaly to a human-review console, every event landing in the lakehouse. This is the phase whose no-AWS core I built and tested.

Databricks courses: Single-Agent Apps (02) + GenAI Deployment & Monitoring (05)

Databricks (managed)

  • Ingestion + features: Delta Live Tables / Spark Structured Streaming + Auto Loader
  • Online features: Databricks Feature Store
  • Agent: single-agent app, UC functions as tools, models-from-code, MLflow tracing
  • Serving: Model Serving endpoint
  • Observability: inference tables + Lakehouse Monitoring

Harbormaster (AWS-native)

  • Ingest: AISStream / replay → Fargate ingestor → Kinesis Data Streams
  • Features: Managed Flink (KDA) keyed windows → Feast / DynamoDB; Firehose → S3/Iceberg
  • Front door: ECS Fargate GeoTrace, deterministic STAGD/TGARD/S-KBM + a HeuristicPlanner (no LLM), POST /v1/score-ais
  • HITL: Postgres review queue + Streamlit console
  • Observability: Prometheus /metrics + cost_tracker + OTel/X-Ray; Iceberg event landing = the inference table

Phase 2

Change Data Capture (CDC)

Stream operational-database changes into the lakehouse so the system of record stays consistent. Databricks reads its own Change Data Feed; Harbormaster captures Postgres changes with Debezium.

Databricks concept: CDF + Auto Loader + apply_changes

Databricks (managed)

  • Delta Change Data Feed (CDF)
  • Auto Loader for incremental ingest
  • DLT apply_changes for SCD upserts
  • Unity Catalog lineage across the change stream

Harbormaster (AWS-native)

  • RDS Postgres → Debezium CDC → Kinesis
  • Firehose → S3/Iceberg as the change sink
  • Iceberg MERGE for SCD-style upserts
  • Glue Catalog lineage; lakehouse is the system of record

Side-by-side stack

CapabilityDatabricksHarbormaster (AWS)

Transport(internal)Kinesis

Why Debezium. Databricks' CDF only captures changes to Delta tables it owns. Harbormaster needs to capture changes from an external operational Postgres, which is exactly Debezium's job (log-based CDC off the write-ahead log).
Phase 3

Lake + model promotion (training & backfill)

Distributed batch over historical AIS, GPU training of the heavy models, and a disciplined promotion gate. Databricks does this on Spark + MLflow; Harbormaster splits it across EMR, MSI, W&B, and SageMaker.

Databricks courses: ML at Scale (01) + Advanced ML Operations

Databricks (managed)

  • Distributed Spark training; Hyperopt HPO
  • Batch inference via spark_udf / pandas UDFs
  • MLflow Registry + @champion alias
  • Feature Store point-in-time joins
  • champion/challenger A/B

Harbormaster (AWS-native)

  • EMR transient Spark backfill of MarineCadastre → Iceberg (auto-terminate)
  • GPU training on MSI Slurm + Apptainer (PyTorch, PhysicsNeMo, Pi-DPM)
  • W&B lineage + SageMaker Model Registry promotion
  • Point-in-time feature export from Iceberg
  • Promotion gate: shadow → canary, auto-rollback on error budgets

The biggest split. "No GPU on AWS" is a hard rule: GPU instances are the fastest way to blow the $75 cap, and MSI provides GPUs free. AWS serves; MSI trains. The cost is a model-promotion boundary to manage.
Phase 4

Drift → HITL → RL flywheel (evaluation & monitoring)

Catch drift, route ambiguous cases to humans, and turn the verdicts into training signal. Databricks uses mlflow.genai judges and a review app; Harbormaster uses statistical gates, a Streamlit console, and a preference-data flywheel.

Databricks course: Agent Evaluation (04)

Databricks (managed)

  • mlflow.genai judges: built-in / guideline / custom / trace-based
  • Offline + online eval
  • Review app for human feedback
  • Lakehouse Monitoring for drift

Harbormaster (AWS-native)

  • MIRROR holdout statistical gate: AUC, CRPS, calibration ratio ∈ [0.8, 1.2]
  • Reward-hacking probe = custom/trace judge
  • SageMaker Model Monitor + Evidently + custom PSI/KS + calibration drift
  • Streamlit HITL = review app; verdicts → preference triples → DPO flywheel

Deliberate. The score is judged statistically, never by an LLM. Statistical gates are reproducible and cannot hallucinate a verdict; the LLM (Bedrock) is confined to explanation.
Phase 5

Multi-tenant, explanation & FDE

Open the platform to multiple tenants, add a natural-language explanation layer, and scale to zero between bursts. Databricks leans on Foundation Model APIs + Vector Search; Harbormaster uses Bedrock (explanation only), the corridor graph, EKS + KEDA, and row-level tenant isolation.

Databricks course: Retrieval Agents / RAG (03)

Databricks (managed)

  • Foundation Model APIs / ai_query
  • Vector Search + managed embeddings
  • Provisioned throughput serving
  • Unity Catalog multi-tenant governance

Harbormaster (AWS-native)

  • Bedrock explanation over deterministic reasons (never the score) + faithfulness guardrail judge
  • Domain retrieval = the corridor graph (RDP + HDBSCAN + NOAA ENC) + optional vector store
  • EKS + KEDA scale-to-zero; per-region SageMaker async MME
  • tenant_id row-level isolation (IAM + Glue + Iceberg)

Synthesis

Five deliberate divergences

Each is a place where I know the managed Databricks default and chose a different AWS-native path for a reason worth defending in an interview.

1 · Cut MLflow (Databricks' core) for W&B + SageMaker Registry. Avoids three-registry sprawl and uses AWS-native promotion gates. The cut is documented, not accidental.
2 · Strip the LLM from the scoring path. Deterministic kernels + a HeuristicPlanner. The agent courses are LLM-centric; Harbormaster's "agent" is deterministic for cost and defensibility. Bedrock only explains, it never computes a score.
3 · Kinesis + Managed Flink, not Spark Structured Streaming or DLT. Right tool for high-volume, keyed AIS telemetry with true event-time state and watermarks.
4 · ECS Fargate, not a managed Model Serving endpoint (and not EKS until Phase 5). The EKS control plane alone is ~$73/mo and would consume the entire $75 cap; Fargate scales to zero cheaply.
5 · Statistical judges (AUC / CRPS / calibration ratio, plus the unbounded hard-physics invariant), not LLM-as-judge for the score. Defensible, reproducible, no hallucinated verdicts.
The senior signal. Knowing both lets me say "I evaluated the managed pattern and chose the self-hosted equivalent because X." The divergences are the talking points; the managed patterns are the reference implementation.
