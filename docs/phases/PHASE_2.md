## Goal
Close the CDC resume gap with a genuinely production-shaped change-data-capture pipeline: analysts edit `vessels` / `watchlist` / `sanctions_flags` in Postgres (via the Streamlit console and the serving API); Debezium captures the changes over logical replication (`wal_level=logical`, pgoutput); an idempotent, LSN-guarded consumer applies them to the online stores (Feast/DynamoDB + Redis invalidation) and appends every delivered event to an Iceberg `cdc_audit` table. The serving scorer reads watchlist state ONLY from the online store, never from Postgres; CDC is what keeps it fresh. Acceptance: flag-to-scored within ~5 s, replay-safe, restart-safe, delete-safe, and slot-lag alerting. Two drills produce war stories P1 (replication-slot bloat) and P2 (duplicate events after a consumer restart).

## Decisions locked for Phase 2
- Kafka default = **Strimzi on a local kind cluster** ($0, the primary demo and CI-adjacent path). **MSK Serverless is an on-demand AWS showcase only**: at ~$0.75/cluster-hour it is a budget killer if left up (~$550/mo), so it exists as a Terraform module behind `enable_phase2`, applied for a demo window and torn down after. The Phase 0 nightly teardown Lambda already sweeps orphaned MSK.
- Kafka itself is Strimzi-managed locally; Debezium Kafka Connect runs as a plain Deployment of the official `quay.io/debezium/connect` image in the same kind cluster (decided at 2.6: the Strimzi `KafkaConnect` CR path requires an image build + in-cluster registry that buys nothing on a single-node throwaway cluster). On AWS it is an ECS Fargate service running the same image (EKS stays deferred to Phase 5 per the Phase 1 decision; the control plane alone would eat the $75 cap).
- Redis on AWS = a tiny **containerized Redis on Fargate** with Cloud Map DNS. ElastiCache is the production answer and is documented as such, but at ~$9-12/mo always-on it loses to a demo-window container under the $75 cap. Same trade-off shape as Phase 1's ALB -> API Gateway HTTP API decision. Locally Redis is a plain container in the kind cluster.
- Online store = the **existing Phase 0 `feast_online` DynamoDB table** (hash `entity_id` S, range `feature_name` S; always-on, on-demand, ~$0 idle). CDC items use `entity_id = str(mmsi)` and `feature_name` in {`vessel_meta`, `watchlist`, `sanctions`}. No new always-on table. Locally the same layout runs on DynamoDB Local.
- Exactly-once is delivered as **at-least-once transport + idempotent sink**, never claimed as transport-level exactly-once: (a) upsert keyed by primary key; (b) a monotonic `last_applied_lsn` guard per (table, pk), enforced in DynamoDB by a conditional expression; (c) Kafka offsets committed only after every sink acks the batch; (d) delete becomes a **soft-delete marker** (`deleted=true` + the delete's LSN) so a replayed older update cannot resurrect a deleted row; the read path treats `deleted=true` as absent; (e) snapshot reads (op `r`) carry LSNs and pass through the same guard, so the snapshot-to-stream transition is duplicate-safe by construction.
- The Iceberg `cdc_audit` table is **append-only at the transport level, deliberately including redeliveries**, with an `applied` boolean recording the guard's verdict. Replaying the topic therefore duplicates audit rows (transport truth) while producing zero duplicate online state (state truth); that distinction IS the demo. Local catalog = pyiceberg + SQLite catalog + file warehouse; AWS = pyiceberg + Glue catalog + the S3 lake bucket (Athena-queryable).
- Postgres stays the system of record; the publication is created explicitly by our DDL (`publication.autocreate.mode=disabled`), REPLICA IDENTITY FULL on the three tables (tiny tables; full before-images make the audit trail complete), heartbeats on (`heartbeat.interval.ms=10000`) so an idle database still advances the slot's `confirmed_flush_lsn` (this is half the P1 story).
- RDS logical decoding is enabled by a new `logical_replication` variable on the existing `modules/rds` (default **false**, so Phase 1 behavior is untouched); when true it attaches a parameter group with `rds.logical_replication=1` (static parameter, reboot at demo-apply time is acceptable).
- Consumer = Python `confluent-kafka` (librdkafka), manual offset commit, all I/O behind injected interfaces so the entire unit suite runs with zero Kafka, zero AWS, zero Docker. On AWS the consumer authenticates to MSK Serverless via SASL/OAUTHBEARER with `aws-msk-iam-sasl-signer-python`.
- Watchlist lookup in the scorer is **fail-open with a counter**: if Redis and DynamoDB are both unreachable the event still scores (without the watchlist reason) and `WATCHLIST_LOOKUP_ERRORS` increments. Availability over freshness, stated, metered, and alarmable; the inverse (fail-closed) would let a cache outage stop all scoring.
- SCOPE GUARD (do not drift): no query routing, no sharding, no consensus implementation. That is Vitess / Multigres territory; Harbormaster consumes Postgres. The Multigres cover-note gap line is updated at gate 2.9, not before.

## Architecture (two planes, one code path)
- **Local plane (default, $0)**: kind cluster -> Strimzi operator -> single-node KRaft Kafka + Debezium Connect (plain Deployment of `quay.io/debezium/connect`) + Postgres 16 (`wal_level=logical`) + Redis + DynamoDB Local, all in-cluster with NodePort mappings; the consumer and the serving API run on the host against those ports. `make cdc-up / cdc-smoke / cdc-down`.
- **AWS showcase plane (demo window only)**: RDS Postgres (Phase 1 module + `logical_replication=true`) -> Debezium on Fargate -> MSK Serverless -> consumer on Fargate -> feast_online DynamoDB + Redis-on-Fargate + Iceberg on the S3 lake via Glue. Slot-lag Lambda + CloudWatch alarm -> the FinOps SNS topic. Requires `enable_phase1=true` (RDS, serving, ECS cluster) AND `enable_phase2=true`; both default false, so `make apply` stays Phase-0-only.
- One consumer/sink code path serves both planes; only endpoints and auth differ via env (`HM_` prefix, 12-factor, mirroring `serving/app/config.py`).

## Data contract and idempotency invariants (encoded as unit tests at 2.4)
1. Apply(e) mutates online state only if `e.lsn > last_applied_lsn(table, pk)`; equal or older LSNs no-op (at-least-once redelivery is free).
2. Offsets commit only after all sinks ack the batch; a crash before commit causes redelivery, which no-ops by (1). Convergence: any interleaving of duplicates and restarts yields the same final online state as exactly-once delivery.
3. Delete writes `{deleted: true, last_applied_lsn: lsn_d}`; a later replayed update with `lsn_u < lsn_d` no-ops by (1), so no resurrection. The Kafka tombstone message (null value after the `op=d` event) is compaction metadata and is skipped.
4. Snapshot rows (op `r`) are upserts under the same guard; re-snapshotting after a slot re-create cannot double-apply.
5. Every delivered envelope lands in `cdc_audit` exactly as delivered (redeliveries included) with `applied` true/false; audit row count == delivered count, applied-true count == distinct effective changes.
