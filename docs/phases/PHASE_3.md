## Goal

Close the training-to-serving gap: a transient EMR Spark job backfills the MarineCadastre history into the Iceberg lake and rebuilds the learned corridor graph; Feast's offline store exposes a point-in-time-correct training-set export; MSI trains Pi-DPM (unchanged, out of scope here) and a one-way, content-addressed manifest carries a finished checkpoint from MSI to S3; W&B records the lineage from raw extract to checkpoint; a SageMaker async Multi-Model Endpoint serves the Pi-DPM head (per-region variants, scale-to-zero); and a CI-gated promotion pipeline (MIRROR-holdout gate -> SageMaker shadow -> weighted canary -> auto-rollback on burn-rate) is the only door a candidate checkpoint can walk through to reach production. Acceptance: a checkpoint promotes only after passing the holdout gate, a clean shadow diff, and a non-burning error budget; bad data fails a Great Expectations suite and blocks training before it ever reaches Iceberg; the EMR backfill auto-terminates under the cost cap. Two drills produce war stories L1 (training-serving skew, caught by shadow, not by holdout) and L2 (canary regression at 25% traffic, caught by burn-rate, reverted automatically).

## Decisions locked for Phase 3

- **Registry: W&B (lineage) + SageMaker Model Registry (promotion approval); MLflow is cut.** Reason (documented, not accidental, per `docs/DATABRICKS_MAP.md`'s "deliberate divergences"): a third registry buys nothing once W&B already owns experiment/lineage and SageMaker Model Registry already owns the AWS-native approval workflow (`ModelApprovalStatus`); three overlapping registries is sprawl, and the master plan's risk list flags exactly this. W&B answers "where did this checkpoint come from" (git SHA, config hash, data fingerprint, run id); SageMaker Model Registry answers "is this checkpoint allowed to serve traffic" (holdout-gate metadata, approval status). Neither needs to also do the other's job.
- **EMR is transient and auto-terminates.** The backfill runs as an EMR Serverless application (submit-and-forget: pay-per-second, no idle cluster to forget about) rather than a persistent EMR-on-EC2 cluster; if EMR-on-EC2 is used instead for a step requiring it, the job flow is created with `keep_job_flow_alive_when_no_steps=false` so the last step's completion tears the cluster down automatically. Either way there is no "remember to terminate it" step; termination is structural, not procedural, matching the Phase 0 FinOps philosophy (guardrails as code, not as a checklist).
- **Everything new lives behind `enable_phase3` (default false), which requires `enable_phase1 = true`.** Phase 3's SageMaker endpoint needs to sit in the Phase 1 VPC/subnets to be reached privately from the ECS serving orchestrator, and the training-set export reuses the Phase 0 lake bucket and Glue catalog that Phase 1's outputs already wire up; `enable_phase3` does not require `enable_phase2` (the CDC plane is orthogonal to the training/promotion path). A base apply stays Phase-0-only and cheap; `enable_phase3=true` is a demo-window toggle, exactly like `enable_phase1`/`enable_phase2`.
- **Canary/shadow promotes the SageMaker Pi-DPM endpoint variant, not the ECS Fargate front door.** `docs/SYSTEM_DESIGN_DECISIONS.md` DR-3 describes the target-state saga ("Argo CD drives the rollout... canary weights step 5/25/50"), written before the Phase 1 decision to defer EKS to Phase 5 (the control plane alone is ~$73/mo, the whole cap). Since only the GPU-heavy Pi-DPM head sits behind SageMaker in Phase 1-3 (the ECS front door orchestrates deterministic kernels and does not itself change on a model promotion), Phase 3 concretizes DR-3's saga using SageMaker's own native production-variant weighting and shadow-testing feature, not Argo CD/Kubernetes. This is not a downgrade of the decision, it is the correct instantiation for what is actually being promoted at this phase: a single model artifact behind a single managed endpoint. Argo CD remains the right tool for the day the whole front door lives on EKS (Phase 5); DR-3 is updated with a forward-reference to this paragraph, not rewritten.
- **The holdout gate reuses MIRROR's real eval code, ported (not cross-repo-imported) into this repo.** `~/code/mirror` is a separate research repo on a different release cadence; Harbormaster's existing pattern for external reuse is to vendor/port the relevant logic (the same choice Phase 1 made for the GeoTrace agents). `mlops/holdout_gate.py` ports: ROC-AUC scoring in the shape of `mirror/eval/generalization.py`'s `run_abc_study`; CRPS via the `crps_ensemble` shape in `mirror/eval/metrics.py`; and the calibration ratio via the exact formula in `serving/monitoring/drift.py:138` (`mean(err**2) / mean(variance)`, 1.0 = perfectly calibrated). One real gap to close in the port: MIRROR's own `check_calibration_drift` flags drift on a symmetric tolerance band (`abs(ratio - 1) > tol`, default 0.25), not the literal `[0.8, 1.2]` range the master plan specifies; Phase 3 adds an explicit range check on top of the reused ratio computation rather than reinterpreting the existing tolerance band as equivalent (they are close but not identical: tol=0.2 centered on 1.0 gives exactly [0.8,1.2], so the fix is a one-line reparametrization, verified by a unit test, not assumed).
- **The MSI -> S3 checkpoint export is one-way and content-addressed.** Neither pi-grpo's raw `torch.save({"model","cfg"}, path)` convention nor its separate `app/services/checkpoint.py` manifest service (content-addressed `runs/<run_id>/step_<n>/<sha256[:16]>.bin` + `MANIFEST.jsonl`) currently records git SHA, config hash, or data fingerprint; Phase 3 builds Harbormaster's own `mlops/manifest.py`, matching the content-addressed layout but adding those fields explicitly, since nothing upstream populates them today. "One-way" is enforced structurally: there is no code path in this repo that pulls an S3 checkpoint back down to resume MSI training (training-on-MSI stays MSI's own concern, per the master plan's hybrid-plane decision); Harbormaster only ever reads a finished manifest to serve it.
- **The Pi-DPM frozen-scorer contract is genuinely stateless, which is why the async MME shape works.** Per the pi-grpo reuse anchor (`app/components/pidpm_scorer.py`, `PiDpmScorer.log_prob(trajectory: np.ndarray) -> float`): a `(T, >=2)` array in, one scalar out, swappable purely by pointing at a new checkpoint file. No hidden state threads between calls. SageMaker's async-inference model matches this exactly (submit an S3 payload, poll or get notified for the S3 output); no custom session/state management is needed in the endpoint container.
- **Great Expectations gates the EMR job, not just the final export.** Bad data must block training before it is ever written to Iceberg (the master plan's explicit acceptance criterion), so the GE suite runs as the EMR job's first Spark step; a failing suite halts the job (no partial/silent write), not a post-hoc data-quality dashboard that discovers the problem after the lake already has bad rows in it.
- **The learned-corridor rebuild supersedes nothing by default.** Phase 1's static demo artifact (`serving/app/artifacts/corridors.json`, `serving/app/components/corridor.py`, `docs/corridor-detector.md`) stays the frozen, checked-in, always-available default; the Phase 3 EMR job's `corridor_graph_nodes`/`corridor_graph_edges` Iceberg output is an optional swap-in behind its own settings flag, so Phase 1/2 goldens are unaffected unless a demo explicitly opts in (same "empty/false disables, existing behavior wins" convention as `online_table` in Phase 2).

One code path, two planes: local unit/smoke tests run the whole promotion state machine (holdout gate, shadow diff, canary transitions) against fakes with zero AWS; the AWS showcase runs the identical `mlops/promote.py` against a real EMR/SageMaker/Registry demo apply (Arun-run).

## Promotion-pipeline invariants (encoded as unit tests at gate 3.7)

1. A candidate that fails any holdout threshold (AUC below `auc_min`, CRPS above `crps_max`, or `calibration_ratio` outside `[0.8, 1.2]`) never reaches SageMaker Model Registry approval, let alone shadow or canary; the gate is a hard stop, not a warning.
2. Shadow traffic never affects production scores: the shadow variant's output is compared and discarded at the invocation site (a parallel run, per DR-3), never fused into the response the caller receives.
3. Canary weight only ever increases (5 -> 25 -> 50 -> 100) while the burn-rate stays under the configured SLO budget (DR-13); a breach at any step reverts the full weight to the prior champion in one action, never a partial rollback.
4. The EMR backfill always carries an auto-terminate action; a Terraform plan or apply that would create an EMR job without one is a build-time failure caught by a unit/plan check, not a runtime hope.
5. The MSI -> S3 checkpoint export is one-way and content-addressed: exporting never overwrites an existing manifest entry (the sha-addressed path makes this true by construction), and no code path in this repo pulls a checkpoint back down to resume MSI training from S3.

## Cost envelope (us-east-1, $75 cap)

Local plane: $0 (local-mode Spark, fakes/moto for GE/EMR/SageMaker/Registry in unit tests). AWS showcase, demo-window only:
- EMR Serverless backfill: pay-per-second compute, no idle cluster; a full 600 GB MarineCadastre backfill run estimated at low tens of dollars for a one-time or infrequent job; a small demo-fixture run is well under $1.
- SageMaker async inference: scale-to-zero between invocations (application auto-scaling minimum capacity 0); pay only for the instance-seconds actually used per invocation plus a small idle-scale-in delay; a demo window of a handful of invocations is a few dollars at most.
- Athena queries for the Feast point-in-time export: ~$5/TB scanned; a bounded export over a demo-sized slice of the lake is cents, not dollars.
- SageMaker Model Registry: no separate charge for the registry itself.
- Glue Data Catalog: within the AWS free tier for the object counts this phase adds.
- W&B: existing free-tier account (already used by pi-grpo), no new cost.

## War stories (produced by this phase, grounded only when hit)

- L1 training-serving skew [PLATFORM / personal-build, CORRECTNESS]: a feature encoded differently offline (the training-set export) than online (the live scoring path) passes the holdout gate, because the gate evaluates the candidate against the same offline-encoded holdout set the skew itself produced, and only shows up once real online-encoded traffic reaches it. Produced by drill 3.8-L1; the fix story is that shadow exists precisely because holdout is necessary but not sufficient; echoes the real precip-standardization normalization bug Arun found and fixed in the PC-RF paper (a mean-subtraction encoding choice that made `coarse_mass` negative on 10 of 16 samples, invisible until the actual physics projection ran).
- L2 canary regression with auto-rollback [PLATFORM / personal-build, MLOPS/RELIABILITY]: a candidate that passes every offline and shadow check regresses only under real canary traffic, a distribution neither the holdout set nor the shadow sampling window happened to cover. Produced by drill 3.8-L2; the fix story is the saga's compensating action (DR-3): burn-rate on the SLO error budget (DR-13) triggers an automatic, full, one-step revert to the prior champion weight, so rollback is a tested path exercised in CI, not an incident-time improvisation.
