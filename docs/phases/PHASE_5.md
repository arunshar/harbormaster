## Locked decisions

- **EKS replaces ECS Fargate as the serving front door; nothing else about the serving container changes.** `infra/terraform/modules/ecs_serving/main.tf`'s own header describes today's shape exactly: "The deterministic AIS scorer as a Fargate service, reachable via Cloud Map private DNS... and, through the apigw module, an API Gateway HTTP API. No standing ALB... Scales 1->3 on CPU." Phase 5 moves that same container image (`serving/Dockerfile`, unmodified) into an EKS `Deployment` + `Service`, fronted the same way (API Gateway HTTP API via VPC Link, no standing ALB, matching the Phase 1 cost decision that ruled out an ALB in the first place); the application code inside the container does not change, only its orchestration substrate.
- **KEDA, not a bare HPA.** Since no working KEDA config exists anywhere to inherit, Phase 5 installs KEDA (via the `keda` Helm chart, a new Terraform `helm_release` resource) fresh, and scales the serving `Deployment` on two real KEDA scaler types matching the master plan's "KEDA scale-to-zero on Kinesis/Kafka lag" bullet: the built-in `aws-kinesis-stream` trigger (scaling on the Phase 1 `ais-raw` stream's `GetRecords.IteratorAgeMilliseconds`) and the built-in `kafka` trigger (scaling on the Phase 2 CDC consumer's consumer-group lag, when `enable_phase2=true`). `minReplicaCount: 0` is the scale-to-zero floor KEDA adds over a bare HPA (which cannot go below 1).
- **The EKS control-plane cost risk gets a structural teardown, not a procedural one.** Mirroring Phase 0's nightly teardown Lambda philosophy ("guardrails as code, not a checklist") rather than repeating the "flip back to false after every demo" prose-only discipline Phase 2's MSK Serverless relies on: a new EventBridge scheduled rule (default: 4 hours after any `enable_phase5=true` apply) invokes a small Lambda that force-destroys the EKS cluster and node group regardless of what is running, unless a Terraform-tagged "keep-alive" flag is explicitly set for an extended demo. This is new scope beyond the master plan's original Phase 5 bullet, added here because EKS is the first Phase 1-5 compute surface whose idle cost is not already zero-able by the mechanism (SageMaker/Kinesis/DynamoDB/EMR-Serverless) every earlier phase relies on.
- **Multi-tenant isolation is Postgres row-level security plus per-tenant SageMaker production variants, default-deny.** Every table gains a `tenant_id` column and an RLS policy requiring `tenant_id = current_setting('app.tenant_id')::uuid`; a request with no tenant context set is rejected by Postgres itself (fail-closed, not an application-layer check that can be forgotten), matching `docs/SYSTEM_DESIGN_DECISIONS.md:112`'s existing decision-record language verbatim. The Pi-DPM SageMaker endpoint (Phase 3) gains one `ProductionVariant` per demoed tenant rather than a shared variant, so a tenant's model quality and traffic are isolated at the AWS resource level, not just the database level.
- **Per-tenant SLO tiers reuse `serving/app/slo.py`'s existing `Slo`/`SloResult`/`evaluate` shapes, extended with a `tenant_id` dimension and a real burn-rate calculator, not replaced.** The three tiers (real-time / near-real-time / batch, per the master plan's FDE SLA-doc bullet) are three different `PHASE1_SLOS`-shaped tuples keyed by tenant tier; `evaluate()`'s pure pass/breach signature is unchanged, called once per tenant per measurement window. The new `burn_rate(errors_consumed, budget, window) -> float` function (closing the DR-13 gap identified above) is genuinely new code, ported from the Google SRE multi-window burn-rate formula DR-13 already cites as its source, not assumed to already exist.
- **Per-tenant drift monitoring is Phase 4's `mlops/drift.py` (`check_input_drift`, PSI/KS) called once per tenant partition, not a new drift algorithm.** The mechanism from gate 4.1 is reused verbatim; Phase 5's addition is purely the per-tenant partitioning of the reference/current windows, so a single-tenant population shift shows up in that tenant's own PSI/KS result instead of being averaged into a global distribution that would dilute it below the alert threshold (the master plan's stated Phase 5 acceptance criterion and incident P4).
- **Bedrock is an explanation layer only, never a scoring path, matching `docs/ARCHITECTURE.md:71` verbatim.** A new `serving/app/bedrock_explainer.py` (mirroring `pidpm_client.py`'s injected-client, fail-open convention exactly: disabled by default, `from_settings` degrading to disabled on construction failure) takes an already-computed, already-scored HITL item's deterministic reasons (the existing noisy-OR reason codes: `off_corridor`, `abnormal_gap`, `WATCHLIST_HIT`, etc.) and asks Bedrock to narrate them in one paragraph; the prompt is constructed so Bedrock only ever receives reasons and scores that already exist, never raw trajectory data it could hallucinate a different verdict from, and the HITL item's `score`/`reasons` fields are never overwritten by anything Bedrock returns.
- **The PPO route-optimization service is a labeled stretch, a separate service, never a core platform claim.** Ported from pi-grpo's real `AdaptiveKLController` + the `PpoTrainer`/`PpoConfig` shape (verified above), retargeted from language-model fine-tuning to a small feasibility-constrained patrol-routing policy (reward = coverage minus fuel, action space = the Phase 3 corridor graph's edges, feasibility = the existing S-KBM kinematic gate). It ships as its own optional service behind `enable_phase5_ppo_stretch` (default false, independent of `enable_phase5`), never merged into the core promotion pipeline or counted toward the "closes these resume gaps" claim in `docs/HONESTY.md`.

- EKS control plane: ~$0.10/hour prorated from the $73/mo flat rate; a 2-4 hour bounded demo window is $0.20-0.40. WATCH: unlike every other Phase 1-4 compute surface, this cost accrues whether or not any node or pod is running, which is exactly why gate 5.0 adds a structural (EventBridge + Lambda), not procedural, teardown, rather than repeating Phase 2's "remember to flip the toggle" discipline.
- EKS node group (spot, scale-to-zero via KEDA/cluster-autoscaler): near $0 at idle, a few cents per demo hour at 1-2 nodes.
- Bedrock: per-invocation token pricing; a handful of HITL explanation calls in a demo window is cents.
- Additional SageMaker production variants (per-tenant MME): the existing Phase 3 cost shape, multiplied by however many tenants are actually demoed (kept to 2-3 for the demo, not a large synthetic tenant count).
- PPO stretch service: CPU-only training on a tiny synthetic corridor graph, effectively free; never runs on AWS GPU (no GPU on AWS is a standing platform-wide decision, unchanged here).

A multi-hour Phase 5 demo window (EKS + KEDA scale test, a few Bedrock calls, an RLS drill, a brief PPO smoke) is estimated at $1-3, inside the $75 cap alongside the Phase 0-4 baseline, PROVIDED the teardown guard actually fires; an EKS cluster accidentally left up for a full month at $73 flat plus node-group cost would be this platform's single largest standing budget threat to date, larger than Phase 2's MSK Serverless ($18/day) or Phase 3's idle-GPU audit catch ($530/mo), which is exactly why gate 5.0's guard is structural.

Stays anticipated unless the live measurement actually breaches a tier.
