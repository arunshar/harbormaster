## 1. How to read this book

Harbormaster is built across five phase docs (`docs/phases/PHASE_1.md` through `PHASE_5.md`), two external audit files, three runbooks, and a holistic master plan. Before this book, understanding the platform's real current state meant reading all of them and reconciling their divergences by hand. This book does that reconciliation once, in one place, and links back to the originals rather than duplicating their gate-level text.

There is no branch map to draw. Every phase doc lives on one branch, `phase4-flywheel`, which is the full linear chain: `master -> phase1-serving-slice -> phase2-cdc -> phase3-lake -> phase4-flywheel`. That chain was assembled by rebasing `phase4-flywheel` onto `phase3-lake`'s current HEAD on 2026-07-04, after `phase4-flywheel` had forked from an older point and drifted behind three live-run fixes. The rebase is itself documented (section 2, section 9) because it is exactly the kind of divergence this book exists to reconcile.

Read section 2 first for the current state in one table. Read sections 5 through 10 for the phase-by-phase build record. Read section 11 if you are deciding what to fix next on the SageMaker code specifically. Read section 12 before an interview; it is the fastest way into the war-story catalog. Read section 13 before running a demo window. Read section 14 for what is genuinely still open.

## 3. Honesty framing

`docs/HONESTY.md` is the locked source of truth and this section defers to it rather than restating it. Three points worth holding in working memory while reading the rest of this book:

ESRI (Summer 2023) is real, is a real team with real clients, and shipped the original detector and its AWS MLOps. That work is ESRI's. Harbormaster is a personal extension built after and separate from that engagement, never merged with ESRI code or data, built entirely on public sources (MarineCadastre historical AIS, live AISStream.io traffic, NOAA charts). Every artifact that could be mistaken for employer work carries a label saying otherwise.

Harbormaster closes four things and says so plainly: change-data-capture, streaming, production distributed systems, and MLOps/observability. It does not close two things and says that plainly too: a sharded query router and a consensus implementation, both Vitess/Multigres territory that this platform deliberately consumes managed Postgres instead of reimplementing.

A new dated section in `docs/HONESTY.md` (added 2026-07-04) records that the MLOps closes-claim moved from designed-and-tested to actually-run-against-real-AWS this sprint, and states two things plainly in the same breath: the SageMaker endpoint served a labeled `phase3-demo-standin` model throughout, never a real trained checkpoint, and Phase 2's AWS MSK showcase has never run. Read that section before writing any claim about what "the live run proved."

## 4. Architecture and decision records

`docs/ARCHITECTURE.md` lays out the hybrid topology in four sections: the topology diagram itself, plane responsibilities, key tradeoffs, and cost/provider posture. The one-sentence version: train on MSI (free H100s, off-cloud), serve on AWS, two planes and one code path, because local fakes and injected I/O prove the same logic the AWS demo windows exercise for real.

`docs/SYSTEM_DESIGN_DECISIONS.md` holds 16 decision records (DR-1 through DR-16), each mapped to a named pattern and a cited source (Kleppmann's *Designing Data-Intensive Applications*, Fowler, Richardson's microservices.io, Nygard's *Release It!*, the Google SRE book, HelloInterview's framework). The records this platform book leans on most:

- **DR-3** (model-promotion pipeline as a saga): shadow, then canary at 5/25/50/100, with burn-rate-triggered auto-rollback as the compensating transaction. Concretized in Phase 3 using SageMaker's native production-variant weighting rather than Argo CD, since only the Pi-DPM head sits behind a managed endpoint through Phase 4; DR-3 carries a forward-reference to Phase 5, where a real front door on EKS would make Argo CD the correct tool again.
- **DR-13** (SLOs, error budgets, burn-rate auto-rollback): the Google SRE multi-window burn-rate pattern. Phase 3's `mlops/promote.py` accepted a `burn_check` callable citing this decision from the start, but no real implementation existed anywhere in the repo until Phase 5's gate 5.4 (section 10). Every burn_check supplied through the live W2 promotion run was an injected fake or a hardcoded `False`. This is the platform's clearest example of a decision record describing a mechanism before the mechanism was built, caught only when Phase 5's own fact-check went looking for it.
- **DR-9** (Kinesis shard / hot-shard design) and **DR-1/DR-2** (CDC pipeline, idempotent consumer): the two structural decisions Phase 1 and Phase 2 instantiate directly.
- **DR-4** (CQRS, read models versus write model): Postgres is the write model for the registry and operational state; Feast, DynamoDB, and a Redis hot cache are the read models; the CDC pipeline is what connects the two. This is the same split DR-1's logical decoding exists to serve, named as its own pattern because the write-side and read-side shapes genuinely diverge.
- **DR-6** (backpressure and the `P_phys` cheap gate): Flink's event-time windows and watermarks let backpressure propagate naturally when a downstream stage slows; the cheap physical-plausibility gate runs first and sheds or short-circuits anything that cannot be anomalous before the expensive model is ever consulted. Phase 1's entire feature job is built around this ordering.
- **DR-7** (multi-tenant isolation as a bulkhead): every record and request carries a `tenant_id`; access is default-deny, so a request with no matching tenant grant is rejected outright rather than silently scoped to nothing. Phase 5's row-level-security design (section 10) is this decision record's first real implementation, not a new idea introduced at that phase.
- **DR-16** (the write/read consistency tradeoff, PACELC): the platform accepts eventual consistency between the Postgres write model and the online-store read model on the hot serving path, and reserves strong consistency for the few places that actually need it, such as reading a just-promoted model's metadata directly from the registry rather than through the online store.

`docs/DATABRICKS_MAP.md` documents the platform's deliberate divergences from an equivalent Databricks-native build, table by table. The one worth repeating here since it recurs throughout Phase 3: "MLflow Registry + `@champion` alias" (Databricks) maps to "W&B (lineage) + SageMaker Model Registry (promotion). Harbormaster cut MLflow on purpose" (Harbormaster), avoiding three-registry sprawl. The audit's HM3-AUDIT-14 finding (section 11) re-affirmed this cut is still correct even as AWS's own example repo increasingly centers managed MLflow, citing pinned pricing (~$438/mo at 24/7, no scale-to-zero) as the reason it still loses to the two-owner split under this platform's cost cap.

## 5. Phase 0: foundations

Deployed live in AWS since 2026-07-03 and unchanged since: VPC and networking, S3 and DynamoDB state stores, the $30 soft / $75 hard FinOps budget pair, cost-anomaly detection reusing the account's existing single service-level monitor, and a nightly teardown Lambda that sweeps orphaned resources and posts a cost summary.

The FinOps design is deliberately guardrails-as-code rather than guardrails-as-checklist. The soft budget alerts at $5, $15, and $25 actual spend plus a $30 forecast, giving early warning long before anything is at risk. The hard budget's breach action is what makes the cap real rather than aspirational: at $75, a Budget Action attaches an IAM-deny policy directly to the platform role, which stops any new resource creation immediately, no human step required between the threshold crossing and the block taking effect. This is also the platform's own first war story (P7, still anticipated): a deny action stops new spend, it does not stop spend already in flight, so a hard cap still needs a structural actuator like the teardown Lambda alongside the deny policy, not the deny policy alone. Cost-anomaly detection reuses the account's pre-existing single Default-Services-Monitor rather than creating a second one, since AWS allows only one dimensional service monitor per account and a second attempt fails outright. The nightly teardown Lambda runs on an EventBridge schedule, scales any running services to zero, stops the Managed Flink application if one is running, sweeps orphaned EMR or MSK resources, and posts a summary to the same SNS topic the budget alerts use. Its dry-run mode was exercised and passed at $0 month-to-date spend before ever running in its real, resource-mutating mode.

This is the only plane that runs continuously; every other phase's AWS surface is a bounded demo window, toggled on and off via Terraform variables, and torn down before the window closes, never left running between sessions.

## 6. Phase 1: streaming and serving

**Goal.** A genuinely production-grade vertical slice: recorded AIS replay through Kinesis, Managed Flink feature computation plus a cheap physics gate, an async score call to a deterministic ECS Fargate front door, an anomaly landing in a Streamlit HITL console, every event also landing in S3/Iceberg, all under the $75 guardrails. Nine sub-gates (1.0 through 1.9), each with a unit test, a smoke test, and a checksum; nothing proceeds past a red gate.

**Locked decisions.** Serving compute is ECS Fargate, not EKS: the GeoTrace container runs as-is, scales to zero, costs about $5/month, while an EKS control plane alone runs about $73/month and would eat the entire cap (this exact tradeoff resurfaces, resolved differently, in Phase 5). The live-scoring path strips GeoTrace's LLM planner entirely: a deterministic `HeuristicPlanner` and a new `POST /v1/score-ais` endpoint run the same deterministic agents (a space-time reasoner, a gap detector, a rendezvous finder, a kinematic validator) at zero LLM cost. AIS input is replay-first from a recorded MarineCadastre fixture, with live AISStream.io ingestion behind a feature flag added at gate 1.9.

**Gates.** The service scaffold and CI (1.1), the GeoTrace serving adaptation with a real Postgres-backed HITL queue replacing an in-memory stub (1.2, plus a corridor-deviation detector add-on using a frozen GTRA-style graph built from NOAA charts), the eight Terraform modules for Kinesis, Firehose, RDS, the ECS cluster and serving/ingestor services, and Managed Flink (1.3), the replay ingestor (1.4), the Flink feature job computing per-vessel kinematics and a cheap physical-plausibility gate (1.5), the Streamlit HITL console (1.6), observability and SLOs (1.7: score success 99.9%, p95 under 300ms on the kernel path, replay-to-HITL p95 under 10 seconds), the end-to-end acceptance gate (1.8, gate G8), and the live-AIS toggle plus a teardown drill (1.9). Before any AWS gate ran, the local slice (1.1, 1.2, 1.5) was built and verified offline: 36 tests passed plus one skipped, ruff clean, and a five-dimension adversarial review confirmed nine findings, all fixed before gate G0.

## 7. Phase 2: CDC pipeline

**Goal.** Close the resume-critical CDC gap with a genuinely production-shaped pipeline: analysts edit vessel, watchlist, and sanctions data in Postgres; Debezium captures the changes over logical replication; an idempotent, LSN-guarded consumer applies them to the online stores (Feast/DynamoDB plus a Redis cache) and appends every delivered event, including redeliveries, to an append-only Iceberg audit table. The scorer reads watchlist state only from the online store, never from Postgres directly; CDC is what keeps it fresh. Acceptance: flag-to-scored within about 5 seconds, replay-safe, restart-safe, delete-safe, with slot-lag alerting.

**Locked decisions.** Kafka defaults to Strimzi on a local kind cluster at zero cost; MSK Serverless is an on-demand AWS showcase only, since at $0.75 per cluster-hour it would run about $550 a month if left up, so it lives behind a Terraform toggle, demo-applied and torn down. Debezium Connect runs as a plain Kubernetes Deployment of the stock image rather than a Strimzi-managed `KafkaConnect` custom resource, since that path needs an in-cluster image registry that buys nothing on a single-node throwaway cluster. Exactly-once is delivered honestly as at-least-once transport plus an idempotent sink: a monotonic `last_applied_lsn` guard enforced by a DynamoDB conditional expression, Kafka offsets committed only after every sink acknowledges, and deletes represented as soft-delete markers so a replayed older update cannot resurrect a row. The Iceberg audit table is deliberately append-only at the transport level, including redeliveries with an `applied` boolean recording the guard's verdict; replaying the topic duplicates audit rows while producing zero duplicate online state, and that distinction is the actual demonstration.

**Gates.** Ten sub-gates (2.0 through 2.9): toggle scaffolding, the registry schema and write path, the online watchlist read path in the scorer, Debezium envelope parsing and connector configuration, the LSN-guarded applier itself, the three sinks (DynamoDB, Redis invalidation, Iceberg audit), the consumer service plus a full local Strimzi stack, the AWS Terraform modules, two drills, and the phase-gate acceptance test encoding five criteria.

**The 17-finding adversarial review.** Run the same day as code-complete, this review found and fixed a dense cluster of real gaps: a snapshot-consistent-point race that could lose a spanning transaction's update (fixed with a floor-guard LSN of zero); an implicit cross-store atomicity assumption in the original invariant list, honestly rescoped rather than solved; no error taxonomy for a poison event, which would have crash-looped the consumer (fixed with a typed content-error class that audits and skips); poison IDs that were themselves mintable (closed at the API and DDL layer); Redis invalidation that only fired on definitively-applied changes, missing a crash-between-write-and-invalidation window (fixed by invalidating on every delivered event, not just applied ones, since a DEL is idempotent); a watchlist lookup running on the event loop, so a partitioned cache could stall every request (moved to a worker thread with a fast fail-open timeout); and, most notably, a real latent Phase 1 defect this review surfaced by accident, the serving task definition never actually injected `DB_SECRET_ARN`, so the HITL Postgres backend had been silently degrading to an in-memory stub on AWS the entire time. The remaining findings closed IAM/networking gaps (a slot-lag Lambda missing VPC endpoints for Secrets Manager and CloudWatch, a null parameter-group fallback that would orphan a custom group), a chicken-and-egg Terraform ordering bug in the two CDC ECR repos, a Terraform version floor bump, a vacuous acceptance criterion that could pass with zero events consumed, and a drill that could have adopted a leftover replication slot or touched the production slot by accident (fixed with a dedicated drill-only slot).

**The local-stack run.** Bringing the stack up for real surfaced one more genuine finding: Strimzi 0.45.0 crash-loops on a fresh kind v0.32 cluster because Kubernetes 1.36 added a new `emulationMajor` field to its `/version` response that the bundled fabric8 client, built for an older Kubernetes API surface, cannot parse. A pinned operator against an unpinned cluster is half a pin; the fix bumped Strimzi to 1.1.0 and migrated the Kafka manifests to the newer CRD API version it requires. With that fixed, the envelope fixture was re-recorded from live topics (identical census: 8 change events, 1 tombstone, 2 skips, 7 applied, 1 guard-rejected), the insert-to-online smoke passed at 0.57 seconds against a 5-second target, and all five acceptance criteria passed green in one 33.5-second `make cdc-e2e` run. Suite after this run: 180 passed, 9 skipped.

**What remains.** The AWS showcase itself. It has never run. This is stated here as plainly as it is stated in `docs/HONESTY.md`'s new section and in `README.md`'s phase table: Phase 2 is local-stack accepted, not AWS-verified, and no claim anywhere in this platform's documentation should read otherwise.

## 8. Phase 3: lake and promotion

**Goal.** Close the training-to-serving gap: a transient EMR Spark job backfills MarineCadastre history into the Iceberg lake and rebuilds a learned corridor graph; Feast's offline store exposes a point-in-time-correct training-set export; a one-way, content-addressed manifest carries a finished checkpoint from MSI to S3; a SageMaker async endpoint serves the Pi-DPM head with scale-to-zero; a CI-gated promotion pipeline, holdout gate through shadow through weighted canary with burn-rate auto-rollback, is the only door a candidate checkpoint can walk through to reach production.

**Locked decisions.** The model registry is split deliberately: W&B owns lineage (where did this checkpoint come from), SageMaker Model Registry owns approval (is this checkpoint allowed to serve traffic), and MLflow is cut as unnecessary sprawl once those two already cover both questions. EMR is transient by construction, not by procedure, auto-terminating structurally rather than relying on anyone remembering to tear it down. Canary and shadow promote the SageMaker endpoint variant specifically, not the ECS front door, since only the Pi-DPM head sits behind a managed endpoint; this is DR-3's saga concretized for what is actually being promoted at this phase, with a forward-reference to Phase 5 rather than a rewrite. The holdout gate ports MIRROR's real evaluation code (AUC, CRPS, a calibration ratio), closing one real gap in the port: MIRROR's own calibration check uses a symmetric tolerance band, not the literal [0.8, 1.2] range this platform specifies, so Phase 3 adds an explicit range check on top of the reused formula rather than assuming the two are equivalent.

**Gates.** Ten sub-gates (3.0 through 3.9): toggle scaffolding, a Great-Expectations-style data-quality gate on the MarineCadastre extract, the transient EMR backfill into Iceberg with corridor-graph derivation via a from-scratch RDP simplification and HDBSCAN clustering, the Feast/Athena offline store and point-in-time export, the checkpoint manifest, the W&B lineage adapter, the SageMaker async endpoint with scale-to-zero, the registry and promotion state machine, two drills, and the phase-gate acceptance test. Real findings folded in along the way: this development machine has no JVM, so every real transform lives in pure pandas/NumPy/scikit-learn functions with the PySpark wrapper around them left honestly untested locally; Feast 0.64 has no native Iceberg offline store but does have a contrib Athena store, which became the real integration point; and the SageMaker model is single-region, not "per-region" as an earlier plan draft implied, since Harbormaster's entire footprint is us-east-1.

## 9. Phase 4: drift, HITL, RL flywheel

**Goal.** Promote `docs/phases/PHASE_4_SKETCH.md`'s three locked design decisions, a drift taxonomy, a preference-triple schema, and a reward-hacking probe, into executable gates. All local-plane: pure functions, injected I/O, fakes; the one piece of Terraform is plan-verified only, never applied.

**The flywheel's shape.** Three drift detectors answer three different questions, because a single signal cannot: input drift (a numpy-only port of MIRROR's PSI and KS statistics), calibration drift (re-calling, not re-porting, the same calibration ratio and [0.8, 1.2] band the Phase 3 promotion gate already enforces, so "drifted out of band" means "no longer the model the gate approved"), and concept drift, which has no ground-truth signal in production anomaly detection and is therefore honestly proxied: a rising volume of near-threshold, high-uncertainty traces cross-validated against the rate at which real human reviewers actually disagree with the model. A decision table routes each combination: input drift alone logs and does nothing; calibration drift routes to plain supervised retraining, the roughly 90 percent case, since labeled data already exists for it; only a rising disagreement rate routes to the preference-triple pipeline and a DPO/GRPO candidate. Every preference triple carries a `hard_violation_in_either_arm` flag so the platform's central physics claim, that a 5.0-weighted unbounded penalty term prevents a kinematically-impossible trajectory from ever winning a preference comparison, is auditable on every triple rather than merely asserted.

**Gates.** Eight sub-gates (4.0 through 4.7): toggle scaffolding, the input-drift port with values cross-checked against MIRROR's original module before being trusted, the calibration-drift watch, the concept-drift proxies and decision table, the preference-triple builder with its own storage and a real HITL-data safety concern (below), the reward-hacking probe wired additively into the promotion state machine, a drift-alerting Lambda authored and plan-verified but never applied, and two drills plus the phase-gate acceptance test.

**The reward-hacking probe's own story, told in full.** The probe's original design blocked a candidate exactly when its mean reward rose and its hard-violation rate also rose. A same-day, second adversarial review attacked the probe itself and found a real gap: a "boundary-riding soft-term hacker," a synthetic candidate that rides the violation-count threshold exactly on most samples while smuggling far more severe violations at a count matched to the honest baseline, so the rate comparison never moves, and wins the reward comparison purely by inflating the softer terms. Under the original, rate-only condition, this candidate promoted with `blocked=False`. Recorded first as a strict expected-failure test asserting the probe should block it, then fixed the same day with the smallest possible change: block when the mean reward rises and either the violation rate rose or the mean hard-term magnitude itself degraded below baseline, no new fields, no new thresholds. The predicted blast radius held exactly: one existing test's expected outcome flipped, correctly, and the expected-failure test converted into a genuinely passing one, its marker removed. Full suite after the fix: 375 passed, 9 skipped.

**The 4-lens adversarial review's other findings.** The same review found that `mlops/preference_data/`, where real HITL trace IDs, MMSIs, and reviewer identities land, had no `.gitignore` entry and no committed fixture despite the gate that produces it promising both, meaning a routine `git add -A` on this personal repository could have committed real operator data to history. Fixed with the ignore rule plus the promised synthetic fixture and smoke target. Separately, the review found that both the disagreement-rate calculation and the preference-triple builder silently exclude a row where a reviewer marked `incorrect` but the model's own implied verdict, from `score` compared against the HITL threshold, does not actually disagree with them, folding this into the same `excluded` count as genuinely ambiguous rows. This is real, intentional, and already tested, a documented data-inconsistency guard rather than a silent miscount, but the gate's own prose in the phase doc never mentioned it, so a reader working from the doc alone would not have expected it. The doc was corrected to match the code rather than the code being changed to match the doc, the right direction of fix when the code's behavior is the one that is actually correct.

**The rebase.** `phase4-flywheel` had forked from an older point on `phase3-lake` and was missing three later live-run fixes to the Flink job. ### The layered-review beat

Findings 01 and 26 (a war story, not an audit finding number) are the clearest single argument in this platform for reviewing more than once. HM3-AUDIT-01 caught a real, dollar-denominated bug before it was ever applied against real AWS. The lesson generalizes past this one alarm: a review that catches a bug class does not certify every other instance of that bug class is now safe, and the review that catches the most is the one that keeps checking after its own first fix lands.

## 13. Operations

Three runbooks cover the AWS-facing side of this platform; none of them is ever run without `make plan` reviewed first, and none of them ever uses `make destroy`.

**`docs/runbooks/PHASE_2_AWS_SHOWCASE.md`.** Package the slot-lag Lambda (`make cdc-lambda-package`); apply infrastructure and ECR repos with no images yet (`make plan`, expect 67 to add, then `make apply`); build and push both images with the platform pinned to `linux/amd64`:

```
aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin "$CONNECT_REPO"
docker build --platform linux/amd64 -f cdc/connect/Dockerfile -t "$CONNECT_REPO:demo" .
docker push "$CONNECT_REPO:demo"
```

Apply again with the image variables set, wait for the two ECS services to stabilize, register the connector in-VPC via `aws ecs execute-command` (Debezium Connect has no public endpoint, on purpose), then run the five acceptance criteria by hand: flag-to-scored under about 5 seconds, a replay against a fresh consumer group producing zero duplicates in the online state hash, a forced Debezium restart losing no change, a delete flipping the online item to its soft-delete marker, and a stalled consumer's slot-lag alarm firing then recovering. Tear down by flipping `enable_phase2` back to false, `make apply` again, never `make destroy`. Cost envelope: MSK Serverless at about $0.75 per cluster-hour, roughly $18 a day if left running by mistake, a three-hour demo landing around $3 to $4. Named threats: the MSK cluster itself, backstopped by the nightly teardown Lambda, and a forgotten Fargate Connect service.

Cost envelope: the same MSK threat as Phase 2, plus Phase 1's RDS, Fargate, and Kinesis costs if left enabled longer than a few hours; the whole window is estimated at $3 to $8 against the $75 cap.

**Cost envelope summary, by phase.** Phase 0: near zero, always on. Phase 1: about $5 a month for the Fargate serving container at rest, low tens of dollars for streaming infrastructure during a demo window. Phase 2: MSK Serverless is the dominant threat at roughly $18 a day if forgotten; everything else is Fargate-container-scale. Phase 3: EMR under a dollar per fixture-scale run; the SageMaker GPU instance is the one that matters, correctly scaling to zero when the two audit-and-self-found dimension fixes are both in place, ~$530 a month if either regresses. Phase 4: zero, entirely local. Phase 5: the EKS control plane's flat ~$73-a-month charge is the platform's largest single standing threat if a cluster is ever left running past a bounded demo window, which is exactly why gate 5.0 adds a structural teardown guard rather than relying on discipline alone.

**Teardown discipline.** Every phase's AWS surface is a boolean Terraform variable, default false. Tearing a phase down is flipping that variable back and running `make apply`, never `make destroy`, which would take the whole account down including Phase 0's always-on foundation. This has now been exercised for real, twice, in this sprint: Phase 3's Part 3 teardown (13 resources, 0 add, 0 change, verified clean afterward) and the routine flip of Phase 2's toggle at the end of every local-stack session.
