# Guardrails first, adversaries always: an experience report on building a production-shaped ML platform solo under a $75/month cap

Written 2026-07-04.

## Abstract

I built Harbormaster, a maritime anomaly-detection platform on AWS, alone, under a self-imposed $75-a-month hard cost cap, to close four resume gaps I named honestly beforehand: change-data-capture, streaming, production distributed systems, and MLOps observability. The unit of contribution in this report is method, not the system itself: cost-as-a-design-constraint, gate discipline, layered adversarial review, and honesty labeling as an engineering practice, demonstrated on one real build. Two internal adversarial reviews found 26 confirmed findings between them, fixed the same day both times. An external audit against AWS's own canonical examples found 19 findings, including a fix-before-demo bug that would have silently cost about $530 a month against a $75 cap. More than 25 war stories are grounded in a live drill or a live AWS run, not a design guess. The platform's own reward-hacking safety probe was red-teamed by its own author and found wanting, then fixed the same day the gap was found. This report claims only what was caught in this build, by these methods; it does not generalize beyond it.

## 1. Introduction

Two constraints are unusual here and shaped nearly every decision: I built this alone, and I built it under a hard dollar cap enforced by code, not a budget line I promised myself I would respect. A third constraint was self-imposed rather than external: a locked honesty contract about where this platform's ideas came from and what it does and does not prove. I worked on a real maritime anomaly detector at ESRI in the summer of 2023, on a real team, with real clients; that work belongs to ESRI. This platform is a personal extension of the same problem, built entirely afterward, on public data, and I say so in every artifact where the distinction could otherwise blur.

Anomaly detection over automatic identification system (AIS) vessel traffic turned out to be a good vehicle for this kind of report. The data is real and continuously arriving, not a static benchmark. The domain has real physics (a vessel cannot exceed its own plausible speed between two positions) that a cheap gate can use to shed the bulk of uninteresting traffic before an expensive model ever sees it. And the anomalies that matter, an unexplained multi-hour silence, a vessel diverging from its shipping lane, are genuinely rare events, which is exactly the condition under which review discipline earns its keep: rare events do not show up in a small test suite no matter how well written.

## 2. Background and provenance

The provenance rule, stated once and applied everywhere: ESRI's work is ESRI's, described here at architecture granularity only, never with its own metrics repeated as if they were this platform's result. Every dataset this platform actually uses is public: MarineCadastre's historical AIS archive, AISStream.io's live feed, NOAA's Electronic Navigational Charts.

The compute plane is hybrid on purpose. Training happens on MSI, a university HPC cluster with free H100 access; serving happens on AWS. This is not a clever architecture choice, it is an honest one: I have no GPU budget on AWS, and pretending otherwise by training there would have blown the cost cap before any serving code existed. Every piece of local logic runs against injected fakes with zero cloud dependency, so the same code path a demo window exercises for real against AWS has already been proven correct offline, at zero cost, before it ever touches a real account.

## 3. System overview

Two planes, one code path. On the training side, MSI runs Pi-DPM training and Weights and Biases experiment logging; a one-way, content-addressed manifest carries a finished checkpoint to S3, with no code path in either direction that pulls a checkpoint back down to resume training remotely. On the serving side, a replay ingestor and, behind a feature flag, a live AISStream.io connection feed Kinesis; Managed Flink computes per-vessel kinematic features (speed over ground, course, heading, distance and gap since the previous fix) and applies a cheap physical-plausibility gate before anything expensive is consulted; a deterministic scoring service on ECS Fargate runs a small set of purpose-built detection agents, a gap detector for unexplained silence, a rendezvous finder for suspicious co-location, a kinematic validator, and a corridor-deviation detector checking a vessel's position against a known shipping-lane graph, with no LLM anywhere in the hot path; anomalies land in a Postgres-backed human-in-the-loop queue with a Streamlit console for review. A separate change-data-capture plane keeps a DynamoDB and Redis online watchlist fresh from Postgres via Debezium and an idempotent, LSN-guarded consumer, so a registry edit, adding a vessel to a watchlist or a sanctions list, reaches the scoring path within about five seconds, replay-safe and restart-safe by construction: replaying the entire change log from the beginning reproduces identical online state even though every redelivery, including ones the system ultimately rejects as stale, is still recorded in an append-only audit trail, a deliberate distinction between transport truth and state truth.

The promotion pipeline and the drift flywheel sit at the center of the MLOps story. A candidate model walks through a holdout gate, a shadow comparison against real traffic, and a weighted canary ramp with burn-rate-triggered auto-rollback, modeled explicitly as a saga: each step has a tested compensating action, so a bad canary weight reverts automatically rather than through an incident-time scramble. On top of promotion sits drift monitoring: three detectors answer three different questions (population shift, calibration, and a proxied concept-drift signal cross-validated against real human disagreement), routing into a preference-data pipeline and a reward-hacking probe that guards against a reinforcement-learning candidate gaming its own reward function.

## 4. Method

**Guardrails before build.** The very first thing built, before any pipeline existed, was the cost cap itself: a hard $75 budget whose breach action attaches an IAM-deny policy directly to the deploy role, a soft $30 budget with tiered alerts, cost-anomaly detection, and a nightly teardown Lambda that sweeps orphaned resources on a schedule. This ordering mattered. A cap that exists as a number in a document is not a cap; a cap that exists as an IAM policy that actually attaches is.

**Plan governance and gates.** Every phase was planned in a dedicated, signed-off document before any code was written, and every sub-phase inside a phase carried a unit test, a smoke test, and a checksum, whether a golden output, a pinned hash, or a Terraform plan count. One commit per gate. Nothing proceeded past a failing gate. When a plan turned out to be wrong once real work began, the plan document was corrected in the same commit as the fix, not left to drift out of sync with the code.

**Two planes, one code path.** Every gate's real logic lives in pure, injectable functions, tested locally at zero cost and zero cloud dependency, before an AWS demo window ever exercises the same code against real infrastructure. This is what made a solo build with intermittent access to a real AWS account tractable: the logic was already proven before the expensive, time-boxed step of proving the wiring around it.

**Honesty rails as an engineering practice, not a disclaimer.** Every artifact that could be mistaken for someone else's work carries an explicit label. A demo-standin model says so in every response it returns. A war story that has not yet been observed live is marked anticipated, not grounded, and stays that way until a real drill or a real run earns it the other label. This discipline is not decoration; it is what makes every other claim in this report checkable.

## 5. The adversarial-review method

Four distinct review layers sit on top of the code, and the honest yield of each is different, which is the argument for running all four rather than trusting any one of them alone.

**Internal multi-lens reviews.** A Phase 1 review confirmed nine findings; a Phase 2 review, run the same day that phase went code-complete, confirmed seventeen, with zero test failures at the time any of them were found. The sharpest of them was a real, live, silent defect: the serving task definition on AWS had never actually injected the database credential the human-review backend needed, so that backend had been quietly falling back to an in-memory stub the entire time, on real infrastructure, caught by accident while reviewing an unrelated part of the same phase. The same Phase 2 review closed a chicken-and-egg deployment ordering bug in two container registries that would have blocked a first apply on a fresh account, and a vacuous acceptance criterion that could have reported success while consuming zero real events. A later Phase 4 review found that a directory holding real, personally-identifiable review data, reviewer identities and trace identifiers from the human-in-the-loop queue, had no ignore rule and no committed fixture despite the gate that produces it promising both, meaning a routine broad `git add` on this personal repository could have committed real operator data to source history; closed the same day it was found. The same review then turned the same discipline on the platform's own reward-hacking safety mechanism and found it wanting (section 6, incident 5).

**Drills that convert anticipated failure modes into grounded incidents.** A stalled change-data-capture consumer's replication-slot lag was allowed to climb from zero to 74,035,000 bytes across five write rounds before the alert fired and the drain was observed; a duplicate-delivery drill showed an unguarded consumer visibly regress a severity score under a specific, adversarial redelivery schedule while a guarded consumer converged to byte-identical state under every schedule tested. Neither of these would have shown up in a normal test run; both required deliberately constructing the adversarial condition and observing the real system's response to it.

**External cross-referencing against vendor canon.** A separate, independent audit ran this platform's SageMaker Terraform line by line against AWS's own canonical example repository and live documentation, and found nineteen distinct findings: two fix-before-demo, seven hardening, six decision-record-only, and four checked and held without change. The lead finding was a dimensionless metric specification that type-checked, plan-checked, and would have silently kept a scale-in mechanism from ever firing, an idle GPU instance running twenty-four hours a day at roughly $530 a month against a $75 cap, with zero exceptions and zero log lines anywhere pointing at the defect.

**Trusting only observed runs.** A CI extras gap was found only by watching the actual pushed GitHub Actions run turn red, not by assuming a fully-populated local development environment matched what a fresh CI runner would need. Live AWS runs found what only reality could: a Python worker subprocess that does not inherit its driver's environment, a managed service's actual runtime being Python 3.9 against a development machine's 3.12, a replication of historical data whose own timestamps silently expired a database record on arrival. None of these were hypothetical; all of them were found because the pipeline was actually run against real infrastructure, not because a reviewer thought harder about the code in isolation.

The layered argument compounds directly in one place. The external audit's fix for the scale-in bug was applied and then verified live, watching real CloudWatch alarm data confirm a genuine zero-instance scale-down. Verifying that exact fix then surfaced a second, structurally identical dimension bug on a sibling alarm the audit itself had never checked, the wake-from-zero path. No amount of re-reading the audit brief more carefully would have caught this; it surfaced only because the first fix was checked live, end to end, rather than trusted the moment the Terraform applied cleanly. A review that catches one instance of a bug class does not certify every other instance of that class is now safe.

## 6. Selected incidents

**Incident 1: duplicate CDC events after a consumer restart.** Symptom: after a forced consumer restart mid-batch, a `--no-guard` drill run visibly regressed a severity score from 0.95 to 0.90 under a specific zombie-rebalance delivery schedule. Wrong first hypothesis: a retry-safe upsert is sufficient for idempotency on its own. Root cause: order matters as much as repetition; an upsert without a monotonic guard can apply an older, redelivered value over a newer one if the redelivery lands after a later legitimate update, which a simple dedup-by-event-ID check would not catch. Fix: a monotonic LSN guard enforced by a database-level conditional write, so a redelivery older than what was already applied is rejected outright. Lesson: idempotency is an ordering guarantee, not just a repetition guarantee, and the adversarial schedule that exposes the gap has to be constructed deliberately, a property test over shuffled delivery orders, not assumed away by a single happy-path replay test.

**Incident 2: training-serving skew invisible to the holdout gate.** Symptom: a candidate model scored a perfect 1.0 AUC on the holdout gate, then failed shadow with a real 0.6056 mean absolute divergence against a 0.05 threshold, once compared against the same batch scored through the actual online path. Wrong first hypothesis: a strong offline evaluation number is sufficient evidence a candidate is safe to promote. Root cause: the holdout set is built from the exact offline feature encoding pipeline the skew itself came from, so the gate is structurally blind to a mismatch between offline and online encodings; it cannot see what it was never given the information to see. Fix: none needed at the gate itself; the fix is architectural, running a shadow comparison against real online traffic as a second, independent check that answers a different question than the offline gate can. Lesson: an evaluation gate can only be as good as the population it is evaluated against, and an offline gate's population is, by definition, never the same population a live system actually serves.

**Incident 3: a deterministic routing layer conditioning on input shape, not values.** Symptom: a documented, planted anomaly reached the scoring service successfully but never appeared in the human review queue, scoring zero reasons every time. Wrong first hypothesis: the anomaly detector's own severity threshold is tuned too conservatively for this case. Root cause: the routing layer only adds the relevant detector to its execution plan once an event's history length reaches a minimum threshold, and the upstream feature job was only ever supplying a single prior data point, so the detector was never in the execution plan at all, regardless of any threshold tuning. Confirmed by constructing the exact expected payload directly and observing it correctly trigger with sufficient history. Fix: changed the upstream feature job's retained state from a single prior point to a rolling window. Lesson: "this input is being scored successfully" and "this input is being evaluated by the detector that matters" are different claims, and a routing layer that conditions on an input's shape rather than its values can make an entire code path permanently dead without any individual request ever failing or looking wrong in isolation.

**Incident 4: an audit finding on paper and a live-verified fix are different claims.** Symptom: a freshly-deployed SageMaker endpoint reported healthy and scored real invocations correctly, with zero errors, zero warnings, and zero log lines anywhere hinting that anything was wrong with its autoscaling configuration. Wrong first hypothesis, implicit in the original code: if the apply succeeds and the autoscaling policies exist with no errors, scale-to-zero will eventually happen on its own. Root cause: the target-tracking metric specification omitted a required dimension the underlying metric is actually published under, so the backing alarms sat permanently in an insufficient-data state, structurally unable to ever fire, not merely slow to react. Fix: applied the audit's recommended dimension addition, then verified live, reading the real alarm state transition out of insufficient-data and confirming a genuine scale-down to zero instances. A second, sibling bug then surfaced on the paired scale-out alarm, an extraneous dimension the metric in question does not publish, found by directly comparing the Terraform against AWS's own canonical example and confirmed empirically by querying CloudWatch with and without the extra dimension. Lesson: "the resource exists and nothing errored" and "the mechanism this resource exists to provide actually functions" are categorically different claims, and the gap between them is invisible by design until the monthly bill, unless the fix is checked against a live, observable state transition, not just a clean Terraform apply.

**Incident 5: a safety probe that lost to its own designer's red team.** Symptom: a synthetic candidate constructed to test the platform's reward-hacking probe raised its mean reward by inflating softer reward terms while riding the exact boundary of the probe's violation-count threshold on most samples and matching the honest baseline's violation count while making the violations themselves far more severe. Under the probe's original design, this candidate promoted with a passing verdict. Wrong first hypothesis, embedded in the original design: a rate-only comparison between a candidate's and a baseline's violation frequency is a sufficient guard against reward gaming. Root cause: a rate comparison is blind to magnitude; a candidate that keeps the same violation count while making each violation much worse never trips a rate-only check. Fix, applied and verified the same day the gap was found: block promotion when the reward rises and either the violation rate rises or the mean violation magnitude itself degrades below the baseline, a change with no new dataclass fields and no new thresholds. The predicted blast radius held exactly: one existing test's expected outcome flipped, correctly, and nothing else moved. Lesson: an unbounded, heavily-weighted penalty term in a reward function is only a real safeguard if something downstream actually audits whether a reward increase came at that term's expense, and a rate-only audit is itself gameable by the exact pattern its own design documentation warned about; red-teaming a safety mechanism with the specific exploit it names as a risk, not just the exploit its first test happened to encode, is what actually tests it.

## 7. Cost engineering results

The platform's always-on plane, the foundational networking and state stores, sits near zero dollars standing at rest. Every other phase's AWS surface is a bounded, Terraform-toggled demo window: applied for a session, exercised, then torn down, verified afterward that nothing remains. A multi-hour demo window covering a live streaming run, a change-data-capture showcase, and a lake-and-promotion pipeline run has been estimated, and separately measured, at three to eight dollars against the $75 cap.

Four named, standing budget threats recur across the phases, each with a structural, not just procedural, mitigation: an always-on managed streaming service left running by mistake (mitigated by a nightly teardown sweep and a demo-window-only toggle); an idle GPU serving instance whose scale-to-zero mechanism silently fails (mitigated, ultimately, by catching and fixing two separate dimension-configuration bugs, one found by external audit, one found by verifying that audit's own fix live); a compute control plane with a flat, non-scale-to-zero cost shape (a later, signed-off phase's largest single risk, mitigated by a new, purpose-built structural teardown guard rather than relying on operator discipline); and a managed experiment-tracking service whose always-on pricing would have made it the most expensive piece of the entire stack, avoided by a documented, deliberate architecture cut in favor of two narrower-scoped, cheaper-to-run tools.

The honest claim here is narrow and specific: the cost cap held because the expensive failure modes were either designed out from the start or caught before they ever accrued real spend. One of the two most expensive potential failures in this build, the idle-GPU scale-to-zero bug, was caught by external review, not by design; that is the direct argument for the layered-review discipline in section 5, not an incidental detail.

## 8. Limitations and threats to validity

This is an n-of-one study: one system, one builder, over a bounded period, and the lessons in section 9 are traceable to specific incidents in this build, not a claim about ML platforms generally. Several evidence artifacts remain uncommitted rather than committed as pinned fixtures: live end-to-end latency figures and a real dollars-per-inference number were named as evidence-capture steps but no transcript backs them at the time of this writing. AWS showcases have run for most, not all, phases: a real-time streaming showcase and a lake-and-promotion showcase both ran live and were torn down clean, while a change-data-capture showcase, the phase most directly tied to this report's central resume-gap claim, has been proven only on a local, zero-cost stack and has not yet run against real AWS. The platform's own catalog of anticipated failure modes are, by definition, predictions rather than experience, and are labeled as such rather than folded into the grounded count. The reinforcement-learning side of the drift flywheel ends at a validated preference-data pipeline; the actual model fine-tune this pipeline feeds has not run. A signed-off, later phase covering multi-tenant isolation, a managed container orchestration front door, and an explanation layer exists only as a plan document; none of its code has been written. Finally, this platform was built with heavy AI pair-programming assistance under my own plan governance, and I disclose that as a load-bearing fact rather than a footnote: the review layers in section 5 exist in part because generated code does not get trusted on its own, and the fact that they caught real, non-trivial defects at every layer is itself part of the honest record, not something to minimize.

## 9. Lessons

Idempotency is an ordering guarantee, not just a repetition guarantee, and the adversarial delivery schedule that exposes the gap has to be constructed on purpose.

An evaluation gate can only be as good as the population it is evaluated against; a purely offline gate is structurally blind to a skew between offline and online encodings, which is exactly why a second, online-facing check has to exist independently.

A routing or fusion layer that conditions on an input's shape rather than its values can make an entire downstream path permanently dead without a single request ever failing in isolation; verify that a code path runs, not just that a request succeeds.

"The resource exists and nothing errored" is a weaker claim than "the mechanism this resource exists to provide actually works," and the gap between the two is invisible by design until it shows up on a bill.

A rate-only or count-only safety check is blind to magnitude; the exploit that gets past it is the one that keeps a rate or count flat while making each individual instance worse.

A review that catches one instance of a bug class does not certify every other instance of that class is now safe; the review that catches the most is the one that keeps checking after its own first fix lands.

A deny policy stops new spend; it does not stop spend already committed, so a hard cost cap needs a structural actuator, like a scheduled teardown, alongside the deny policy, not the deny policy alone.

A cost decision, once made, is not permanent; it is worth re-checking against a vendor's current pricing and posture before repeating it as settled, since the landscape underneath a settled decision keeps moving even when the decision itself does not need to change.

Labeling what has and has not been proven, in the same artifact that makes the claim, is what makes every other claim in a project like this checkable by someone who was not there when it was built.
