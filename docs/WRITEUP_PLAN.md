One story, rendered four ways. Every document draws its sections from this spine and cuts to its audience; nothing outside this spine goes into any document without checking it against a repo artifact first.

## N1. Problem framing (the hook)

A solo engineer rebuilds a production maritime anomaly-detection platform on AWS, honestly, under a $75/month hard cost cap, to close four named resume gaps: change-data-capture, streaming, production distributed systems, and MLOps/observability. The domain is real (live AIS vessel traffic, ~150K ships; ~600 GB MarineCadastre history, 2009–2017). The provenance is handled with a locked honesty contract: ESRI (Summer 2023) shipped the original detector and its AWS MLOps and that work stays ESRI's; Harbormaster is a clearly labeled personal extension on public data, never merged with employer work, and it says out loud what it does NOT prove (no sharded query router, no consensus implementation).

## N2. Method before build (the discipline)

- Guardrails before spend: Phase 0 built the $75 IAM-deny budget action, tiered $5/$15/$25/$30 alerts, cost-anomaly detection, and a nightly teardown Lambda before any pipeline existed.
- Plan governance: a holistic master plan plus an ultra-refined per-phase doc authored and signed off BEFORE execution; any change updates both docs before code; no scope drift mid-phase.
- Gates: every sub-phase has a unit test, a smoke test, and a checksum (golden output, pinned SHA, or Terraform plan count); one commit per gate; nothing proceeds past a red gate.
- Two planes, one code path: heavy training on MSI (free H100s, off-cloud), serving on AWS; local $0 stacks (kind/Strimzi, DynamoDB Local, pure-function fakes) prove the same code the demo-window AWS applies run.

## N4. The centerpiece: the drift -> HITL -> RL flywheel

This is the narrative peak in all four documents. Tell it as a decision discipline, not an RL flex:

1. Drift taxonomy with three different detectors because they answer three different questions: input drift (PSI/KS, ported numerically-verified from MIRROR), calibration drift (the SAME calibration_ratio and [0.8, 1.2] band the promotion gate enforces, so "drifted out of band" means "no longer the model the gate approved"), concept drift (no ground truth exists in production anomaly detection, so it is proxied, stated openly: near-threshold + high-epistemic-variance volume, cross-validated against the HITL disagreement rate).
2. The decision table: input drift alone logs; calibration drift routes to plain supervised retraining (the ~90% case, because labels exist); ONLY rising HITL disagreement routes to the preference pipeline. Proxy 1 rising with proxy 2 flat is explicitly NOT concept drift (the false-alarm row, drill L3, war story P27).
3. Preference triples: HITL operator verdicts and reward-synthesized pairs, ported from pi-grpo's real entry points, with `hard_violation_in_either_arm` computed on every triple so the physics claim ("the 5.0-weighted unbounded hard term means no preference can make an impossible trajectory win") is auditable, not asserted.
4. The reward-hacking probe: before any RL-fine-tuned candidate promotes, block if mean reward rose AND (hard-violation rate rose OR mean hard degraded). Wired into the same promotion state machine, between gate and shadow.
5. The self-inflicted punchline (always include, now with the stronger, fixed ending): the Phase 4 adversarial review attacked the probe itself and found a REAL gap; a boundary-riding candidate that inflates soft/data/pref terms while riding `hard == threshold` exactly, smuggling two severe violations at a matched violation COUNT, promoted with `blocked=False` under the original rate-only condition. Recorded as a strict xfail test asserting the probe SHOULD block. Then fixed: the exact minimal fix proposed (`blocked = mean_up and (rate_up or candidate_mean_hard < baseline_mean_hard)`) landed, the predicted blast radius held exactly (one existing test's outcome flipped, correctly), and the xfail genuinely converted to a passing test, marker removed. The flywheel's safety mechanism was red-teamed, found wanting, and then actually fixed, not just flagged, in the same day. War story P28.

3. External cross-referencing against vendor canon (the sagemaker-deepdive audit): a separate session audited Phase 3's SageMaker Terraform line-by-line against aws/amazon-sagemaker-examples (clone @262e3d5) and live AWS docs, at pinned harbormaster commit b773c15. 19 numbered findings (HM3-AUDIT-01..19): 2 fix-before-demo (both now fixed), 7 hardening, 6 decision-record-only, 4 no-change-validated. Trusting only observed runs: the CI extras gap found by watching the real GitHub Actions run instead of assuming local == CI; the W1 and W2 live-run bugs (P13-P26) found only because the pipelines were actually run against real AWS; the EMR runtime's Python 3.9 vs dev 3.12 (P18) found by pulling the real image; a shared, non-worktree git working directory causing a branch-state hazard, caught before any real damage because a plan diff was treated as suspicious rather than accepted at face value (P23).
