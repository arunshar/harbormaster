Gate G0 on-ramp

Harbormaster AWS setup

The one-time setup that unblocks gate G0 and every AWS gate in Phase 1. It turns the 8-step prose in PHASE_1.md into an ordered, mostly-scripted runbook so your hands-on work is only the console-only account hygiene.

Before you start

Prerequisites

All verified present on this machine. Region is us-east-1 throughout (Budgets and Cost Explorer report there regardless).

  • AWS CLI v2 (aws --version)
  • Terraform >= 1.6 (terraform version)
  • docker (for the Phase 1 container build later)
  • jq (the bootstrap script uses it to build the trust policy)
One idea, in order. Do the five steps in sequence. Each ends with a checksum; do not proceed past a red checksum.
Step 1 · console only

Account hygiene

This is the part that cannot be scripted: it is account-level security you do in the AWS console.

  1. Enable MFA on the root user.
  2. Create an admin identity you will actually use: an IAM user with AdministratorAccess (make an access key for it), or IAM Identity Center (SSO). This is your break-glass identity; you run Terraform as this, not as root and not as the platform role.
  3. Configure the CLI as that admin: aws configure (or aws configure sso).
Never configure the CLI as root. Root access keys bypass the $75 FinOps guardrail (the deny action attaches to the platform role, not root) and are the top AWS security risk. If you already made root keys, delete them once the admin works: IAM → account menu → Security credentials → Access keys.
Checksum. aws sts get-caller-identity shows an ARN ending in :user/<name> or an SSO role, NOT :root; aws configure get region returns us-east-1.
Step 2 · scripted

Platform role + state bucket

Dry-run first (changes nothing, needs no credentials), then run for real.

bash infra/aws/bootstrap.sh --dry-run
bash infra/aws/bootstrap.sh          # confirms each mutating step; --yes to skip prompts

It creates only the two things Terraform cannot bootstrap itself:

  • IAM role harbormaster-platform (trust = your account with MFA; PowerUserAccess + an IAM-management inline policy). It is the platform_role_name in terraform.tfvars and the target the $75 deny action attaches to. Deliberately not your admin identity, so a freeze never locks you out.
  • S3 bucket harbormaster-tfstate-<account-id> (versioned, AES256, public access blocked), the dedicated remote-state store, separate from the data lake.

The script prints the exact backend.tf values to use in step 4.

Checksum. aws iam get-role --role-name harbormaster-platform returns the role; aws s3api get-bucket-versioning --bucket harbormaster-tfstate-<account-id> shows Enabled.
Step 3 · terraform

Phase 0 apply

terraform.tfvars is pre-filled; confirm only alert_email.

make fmt        # terraform fmt -recursive
make validate   # terraform validate (-backend=false, no creds needed)
make plan       # review the resource plan against the $75 cap
make apply      # prompts 'yes'; creates network + state stores + FinOps guardrails
Confirm the alert email. AWS sends a one-time SNS confirmation link to alert_email. Click it, or you receive no budget alerts.
Checksum (gate G0 core). The $75 budget action, the cost-anomaly monitor, and the teardown Lambda all exist; terraform -chdir=infra/terraform/envs/base output shows lake_bucket_name, tf_state_lock_table_name, budget_alerts_sns_topic_arn, spend_freeze_policy_arn, teardown_lambda_name. Enable Cost Explorer in the console (~24h to populate).
Step 4 · you + terraform

Migrate to remote state

Use the values bootstrap.sh printed, confirming the lock-table name from terraform output tf_state_lock_table_name.

  1. In infra/terraform/envs/base/backend.tf, comment out the backend "local" block and uncomment the backend "s3" block, filling the literal bucket + lock-table.
  2. terraform -chdir=infra/terraform/envs/base init -migrate-state (Terraform copies local state into S3).
Checksum. State now lives in the S3 bucket; a fresh make plan shows no drift.
Step 5 · you

Guardrail proof

The final gate-G0 check: prove the cost controls actually bite.

  1. Simulate a small over-spend (or temporarily lower the budget threshold) and confirm the IAM-deny action blocks new resource creation on the platform role.
  2. Run the teardown Lambda with DRY_RUN=true and confirm the SNS cost summary arrives.
Checksum. The deny action blocks new resource creation; the teardown Lambda dry-run logs what it would stop and posts the SNS summary.
